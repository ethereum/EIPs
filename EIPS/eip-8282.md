## Abstract

Introduces two [EIP-7685](./eip-7685.md) request types and corresponding predeploy contracts for [EIP-7732](./eip-7732.md) builders: a builder deposit contract, serving both initial registration and stake top-ups, and a builder exit contract, authorizing full exits by the builder's `execution_address`. Both follow the request-bus pattern of [EIP-7002](./eip-7002.md) and [EIP-7251](./eip-7251.md). Builders no longer onboard through the validator deposit flow, nor exit through the voluntary-exit operation.

## Motivation

[EIP-7732](./eip-7732.md) introduces builders as a separate, staked consensus-layer actor but sources their lifecycle from validator flows: a builder is registered by a validator deposit whose withdrawal credential carries the `0xB0` `BUILDER_WITHDRAWAL_PREFIX`, and exits through a builder branch of the voluntary-exit operation. Dedicated builder request types are preferable:

- **Decoupled registries.** With dedicated types, the actor is explicit from the request type alone: the consensus layer no longer routes deposits by inspecting credential prefixes, and the validator and builder registries are keyed independently.
- **Bounded verification.** A builder deposit's proof-of-possession is verified inline when the deposit is processed, unlike a validator deposit's, which is deferred to the churn-limited `pending_deposits` queue. Carried on the validator deposit request, builder deposits inherit its high per-payload ceiling; a dedicated request type isolates the verification work and caps it at `MAX_DEPOSIT_REQUESTS_PER_BLOCK` per block.
- **Cold-key exit.** [EIP-7732](./eip-7732.md) exits require a signature from the builder's BLS key — the hot key that signs bids. The exit contract instead authorizes exit by the builder's `execution_address`, exactly as [EIP-7002](./eip-7002.md) does for validators via withdrawal credentials.

Builders that must exist at the fork are unaffected: [EIP-7732](./eip-7732.md)'s fork-transition onboarding of builder-credentialed pending deposits is retained; only post-fork onboarding moves to the new contract.

### Deployment

Both contracts are deployed as the [EIP-7002](./eip-7002.md) and [EIP-7251](./eip-7251.md) predeploys were: by a one-time presigned transaction from a single-use deployer account (Nick's method), so the final addresses are derived from those transactions and will be fixed once the runtime bytecode is frozen. Each contract's init code sets its `excess` slot to `EXCESS_INHIBITOR`.

The deployment transactions MUST be included before the fork that activates this EIP. If there is no code at either address once the EIP is active, every block from activation onward MUST be invalid.

### Request queue and system call

Both predeploys follow the [EIP-7002](./eip-7002.md) / [EIP-7251](./eip-7251.md) contract design, reusing their storage layout. There is no ABI: each contract dispatches on the caller and on `calldatasize` alone.

- **System call** — from `SYSTEM_ADDRESS` with empty calldata, at the end of each block: the contract MUST dequeue up to its per-block maximum records (`MAX_DEPOSIT_REQUESTS_PER_BLOCK` / `MAX_EXIT_REQUESTS_PER_BLOCK`, oldest first), return their concatenation as its `request_data`, update `excess` (`excess = max(0, excess + count - target)`, treating `EXCESS_INHIBITOR` as `0`), and reset the per-block count. Records beyond the cap remain queued for subsequent blocks.
- **Write path** — from any other caller with calldata of exactly the contract's input size: the contract MUST validate the request and the value sent (see below), append one record to its queue, increment the per-block count, and emit the accepted record as an anonymous log.
- **Fee getter** — from any other caller with empty calldata: the contract MUST return the current fee without modifying state, and MUST revert if any value is attached.
- Any other calldata size MUST revert.

The execution layer prepends the contract's request-type byte and includes `request_type ++ request_data` in the block requests list, committed via the `requests_hash` ([EIP-7685](./eip-7685.md)). The system call follows the [EIP-7002](./eip-7002.md) rules: invoked as `SYSTEM_ADDRESS` with a dedicated gas limit of `30_000_000` that does not count against the block gas limit; if either contract's system call fails, the block MUST be invalid.

### Request fee

Each request carries a fee, computed exactly as in [EIP-7002](./eip-7002.md):

```
fee = fake_exponential(MIN_REQUEST_FEE, excess, REQUEST_FEE_UPDATE_FRACTION)
```

where `fake_exponential` is the [EIP-1559](./eip-1559.md)-style integer approximation of `MIN_REQUEST_FEE · e^(excess / REQUEST_FEE_UPDATE_FRACTION)`. The fee rises super-linearly while blocks contain more than the target number of requests and decays back to `MIN_REQUEST_FEE` otherwise; it is charged on top of any staked value and left locked in the contract.

Because `excess` starts at `EXCESS_INHIBITOR` and the fee getter reverts while it holds that value, no request can be enqueued until the first system call — i.e. until the activation block.

### Deposit requests

A deposit request is submitted by calling `BUILDER_DEPOSIT_CONTRACT_ADDRESS` with calldata of exactly `184` bytes:

| Bytes | Field |
| --- | --- |
| `0:48` | `pubkey` — 48-byte BLS public key |
| `48:80` | `withdrawal_credentials` — 32-byte commitment (`version` byte + `execution_address`) |
| `80:88` | `amount` — big-endian `uint64`, in gwei |
| `88:184` | `signature` — 96-byte BLS proof-of-possession |

A deposit request serves both a builder's first deposit and subsequent top-ups. The contract MUST reject the request unless `amount * 1 gwei >= BUILDER_MIN_DEPOSIT` and `msg.value >= amount * 1 gwei + fee`; any value beyond `amount * 1 gwei + fee` is retained by the contract and not credited to the builder.

On success the contract queues the 184 input bytes; the dequeued record is the input verbatim with `amount` converted to little-endian, as in [EIP-7002](./eip-7002.md). The `signature` is not verified by the contract — it is carried in the record and verified by the consensus layer.

### Exit requests

An exit request is submitted by calling `BUILDER_EXIT_CONTRACT_ADDRESS` with calldata of exactly `48` bytes: the `pubkey` of the builder to exit. The contract MUST require `msg.value >= fee`; it stakes no value. On success it queues a record of `source_address (20) ++ pubkey (48)`, where `source_address` is `msg.sender`.

Authorization is by `source_address`, as in [EIP-7002](./eip-7002.md): the contract records `msg.sender` verbatim and performs no further check; the consensus layer honours the request only when `source_address` equals the target builder's `execution_address`.

### Consensus layer processing

The consensus layer decodes each dequeued record into one of two SSZ containers, selected by request type:

```python
class BuilderDepositRequest(Container):
    pubkey: Bytes48
    withdrawal_credentials: Bytes32
    amount: uint64  # Gwei
    signature: Bytes96

class BuilderExitRequest(Container):
    source_address: Bytes20
    pubkey: Bytes48
```

A type's `request_data` is the concatenation of the fixed-size SSZ serializations of its records — exactly the bytes the system call returns, in order. `BuilderDepositRequest` is the [EIP-6110](./eip-6110.md) `DepositRequest` without the `index` field.

Detailed state-transition behavior is specified in consensus-specs. In summary:

- A `BuilderDepositRequest` for a `pubkey` not in the builder registry registers a new builder if its proof-of-possession `signature` — over `(pubkey, withdrawal_credentials, amount)` under `DOMAIN_BUILDER_DEPOSIT`, a builder-specific signing domain — is valid; a record with an invalid signature is ignored and its stake forfeited. Deposits are applied immediately, not routed through the validator `pending_deposits` queue.
- A `BuilderDepositRequest` for an already-registered `pubkey` is a top-up: `amount` is credited and the record's `withdrawal_credentials` and `signature` are ignored, as with validator deposits.
- A `BuilderExitRequest` initiates a full exit only if its `pubkey` is an active builder, its `source_address` equals that builder's `execution_address`, and the builder has no pending balance to withdraw. Otherwise the record is ignored — consumed, not re-queued, its fee spent — and must be resubmitted.

### Changes to EIP-7732

- **Deposit routing.** The builder branch of `process_deposit_request` is removed, so a deposit to the validator deposit contract is always an ordinary validator deposit; builders are created and topped up only through `BUILDER_DEPOSIT_REQUEST_TYPE`. Note that a post-fork deposit to the validator contract with a `0xB0` credential mints a validator that cannot withdraw its balance; builder deposits MUST be sent to the builder deposit contract.
- **Fork-transition onboarding.** [EIP-7732](./eip-7732.md)'s one-time onboarding of builder-credentialed pending deposits at the fork is retained, so builders exist from the first slot. This is the only path that onboards builders through the validator deposit contract; the `0xB0` `BUILDER_WITHDRAWAL_PREFIX` is deprecated afterward.
- **Exit routing.** The builder branch of `process_voluntary_exit` is removed, making the voluntary-exit operation validator-only; builders exit only through `BUILDER_EXIT_REQUEST_TYPE`.

## Rationale

- **Two predeploys, two request types.** Mirrors withdrawals (`0x01`) and consolidations (`0x02`): the execution layer needs no new read semantics, and the consensus layer routes by request type rather than by inspecting credentials.
- **One request for deposits and top-ups.** Exactly as the validator deposit contract behaves: the proof-of-possession is checked on a `pubkey`'s first appearance, and later deposits credit stake. A top-up cannot redirect a builder's withdrawals because its `withdrawal_credentials` and `signature` are ignored.
- **Exit by `execution_address`.** A builder's BLS key is hot — it signs bids continuously — so it should not also authorize exits. Routing exit through the cold `execution_address` mirrors [EIP-7002](./eip-7002.md)'s rationale for validator withdrawal credentials, and gives builders a single, well-defined exit authorizer.
- **EIP-1559-style request fee.** The same demand-responsive fee as [EIP-7002](./eip-7002.md)/[EIP-7251](./eip-7251.md) meters submission, together with the per-block caps and the per-deposit stake.
- **Fork-transition onboarding.** Some applications depend on builders existing from the first slot of the fork, which post-fork deposits to the new contract cannot provide. The existing [EIP-7732](./eip-7732.md) onboarding path is therefore retained for the initial builder set.

## Backwards Compatibility

This EIP is additive at the execution layer: it introduces new contracts at previously empty addresses and does not modify the validator deposit contract or the validator request predeploys. At the consensus layer it modifies [EIP-7732](./eip-7732.md)'s builder lifecycle as described in [Changes to EIP-7732](#changes-to-eip-7732); builders onboarded at the fork are unaffected.

## Security Considerations

- **Exit authorization.** The exit contract records `msg.sender` as `source_address` and performs no further check. Because the request carries no signature, the consensus layer MUST initiate an exit only when `source_address` equals the target builder's `execution_address`; otherwise an arbitrary caller could exit any builder.
- **Signing-domain separation.** Builder deposits are signed under `DOMAIN_BUILDER_DEPOSIT`, distinct from the validator `DOMAIN_DEPOSIT`, so deposit proofs-of-possession cannot be replayed across the two classes. (Fork-transition seed deposits, made through the validator deposit contract, are necessarily signed under `DOMAIN_DEPOSIT`.)
- **Replayable deposit records.** A deposit's fields are public in calldata, so a third party can resubmit them for an already-registered builder at an amount it funds itself. Such a replay is a top-up — the credentials and signature are ignored — so it credits stake and redirects nothing.
- **Custodial-split exit standoff.** Exit requires a zero pending balance, every winning bid adds pending balance, and only the `execution_address` can authorize exit. Where the `execution_address` (capital owner) and the BLS key (bidding operator) are held by different parties, an operator that keeps winning bids can indefinitely block the owner's exit. Delegating parties SHOULD retain off-chain control over the operator's bidding.
- **Spam and state growth.** The per-block caps bound the drain rate, not enqueue: the in-state queues can grow across blocks. Growth is gated by value — each deposit locks at least `BUILDER_MIN_DEPOSIT` plus the fee. An attacker submitting valid proofs-of-possession forfeits nothing (the stake remains a withdrawable builder balance), so post-fork onboarding can be delayed behind a FIFO wall of attacker deposits at the cost of locked capital; this is tolerable because the time-critical initial builder set is seeded through the fork transition, not the steady-state contract.
- **Locked funds.** The request fee, any overpayment, and the principal of a first deposit the consensus layer rejects for an invalid proof-of-possession are permanently locked in the predeploy. The execution layer does not verify BLS signatures, so submitters SHOULD verify the proof-of-possession off-chain before broadcasting.
- **Same key as validator and builder.** The registries are keyed by independent request types, so one public key may exist in both, with distinct indices and lifecycles; neither request type can act on the other registry. Builder indices are reused after an exited builder's balance is swept, so clients caching builder state by index MUST account for reuse.
