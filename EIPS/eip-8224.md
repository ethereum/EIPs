## Abstract

This EIP introduces a new [EIP-2718](./eip-2718.md) transaction type - the **counterfactual transaction** - for protocol-native shielded gas funding without requiring the controlling EOA to hold public ETH. Instead of charging gas to `tx.sender` or to some application-specific payer after the call completes, the transaction carries a zero-knowledge proof that authorizes gas payment from a dedicated ETH-denominated **fee note** in a canonical protocol fee-note system.

Validation requires no EVM code execution - only ZK proof verification (bounded cryptographic computation) and fixed storage reads for fee-nullifier and fee-root status - making it compatible with inclusion lists and stateless validation.

This EIP solves the gas-bootstrapping problem for transactions whose first call cannot safely rely on post-execution balances. The Contract Payer Transaction[^contract-payer-tx] is one important companion use case, but not a protocol requirement of this transaction type. A dedicated canonical fee note provides a pre-execution, mempool-safe gas guarantee while preserving privacy. Any portion of that fee note not consumed by gas is intentionally returned as public ETH to `gas_refund_recipient`, making the fee note a bounded ETH exit path as well as a gas funding mechanism. This is what allows arbitrary applications, including third-party privacy pools of other assets, to use the transaction type: the application call can deliver asset `X`, while the fee note provides the ETH bootstrap.

No ETH is left stranded behind a spent fee nullifier: the fee note is fully consumed on execution, with the remainder sent to `gas_refund_recipient`.

## Motivation

### The Bootstrapping Problem

The Contract Payer Transaction[^contract-payer-tx] enables a smart contract account to pay its own gas when called by an authorized EOA. After some initial transaction funds or configures a fresh smart contract account and the account calls `authorize` on the payer registry, the controlling EOA can operate gaslessly forever. No ETH funding transaction to the EOA is needed, so there is no on-chain link between the EOA and the private source of that initial funding.

However, the initial private transaction itself needs gas. This creates a circular dependency:

1. The fresh EOA cannot pay gas - it has no ETH, and receiving ETH from any source creates a traceable on-chain link that defeats the privacy goal.
2. The fresh smart contract account cannot pay gas - it has no ETH until the initial private action executes.
3. The system has sufficient value to pay for gas, but the ETH needed for gas is still locked behind private authorization that has not yet been exercised on-chain.
4. A third-party relayer paying gas leaks metadata (timing, IP, relayer identity) and requires trust in an external party.

The transaction *can* authorize enough ETH to cover gas, but under the current execution model, gas must be escrowed *before* execution begins. The value exists; it is simply locked behind an operation that has not yet been permitted to run.

This EIP is intended to solve that bootstrap problem for legitimate privacy-preserving onboarding and application interaction. Its purpose is to remove the forced on-chain linkage created when a fresh controlling EOA must first receive public ETH solely to pay gas, while preserving ordinary transaction execution semantics once the transaction begins.

The same linkage problem also appears in stealth-address workflows: a fresh stealth recipient can receive assets privately, but still needs a non-linkable way to pay for its first transaction.[^stealth-guide]

### Counterfactual Validation as the Solution

This EIP resolves the circular dependency by separating **application execution** from **gas authorization**. The transaction includes a proof that the sender controls a dedicated fee note that is sufficient to pay up to `gas_limit * max_fee_per_gas`.

The fee note is consumed at the transaction-envelope level, so gas can be paid even if execution reverts. The application call itself still happens in ordinary EVM execution, preserving normal transaction semantics. Because the proof only reveals a nullifier and public transaction parameters, no information about the depositor's identity leaks through the gas payment mechanism beyond what the fee-note system already reveals.

The fee note is therefore not "gas-only" in the narrow sense. Its unused portion intentionally emerges as public ETH at `gas_refund_recipient`. That bounded ETH egress is a feature: it bootstraps the recipient account with ETH and also lets third-party privacy pools of non-ETH assets compose with this transaction type, because only the gas side must be ETH-denominated.

### Relationship to Contract Payer Transaction

The Contract Payer Transaction[^contract-payer-tx] is an optional companion for the smart-account lifecycle use case. Together, the two transaction types provide full-lifecycle privacy for smart contract accounts:

| Phase | Mechanism | Used |
| --- | --- | --- |
| Initial funding | This EIP's counterfactual transaction | Once per account |
| Ongoing operation | Contract Payer Transaction | Every subsequent transaction |

For smart-account bootstrapping, the counterfactual transaction is needed only once - to bootstrap the smart contract account with ETH and register the controlling EOA in the Contract Payer Transaction payer registry.[^contract-payer-tx] All subsequent transactions use standard sponsored transactions with static registry validation.

This also makes the transaction type useful beyond account bootstrapping and beyond ETH-only pools. Any application can be called using a canonical fee note for gas, and a third-party privacy pool can use this transaction type to withdraw some pool-specific asset while the fee note supplies the ETH bootstrap needed to enter the optional Contract Payer Transaction lifecycle if the application chooses to use it.[^contract-payer-tx]

### Compatibility with Partial Statelessness (VOPS)

Under Validity-Only Partial Statelessness (VOPS), this EIP's validation reads are bounded to:

- One storage slot from the canonical fee-note contract for the fee nullifier status.
- One or more known storage slots for the fee-note system's currently accepted Merkle root set.
- Cached canonical verification key data.

A VOPS node would need to retain the storage trie for the canonical fee-note contract in addition to the Contract Payer Transaction payer registry at `0x13`.[^contract-payer-tx] This is a tractable exception of the same class as existing system contracts.

### Counterfactual Transaction

A new [EIP-2718](./eip-2718.md) transaction is introduced where the `TransactionType` is `COUNTERFACTUAL_TX_TYPE` and the `TransactionPayload` is the RLP serialization of:

```
rlp([chain_id, nonce, max_priority_fee_per_gas, max_fee_per_gas,
     gas_limit, to, value, data, access_list,
     fee_nullifier_hash, fee_root, fee_denomination, gas_refund_recipient,
     proof_data, y_parity, r, s])
```

The fields follow the same semantics as [EIP-1559](./eip-1559.md) with the following constraints:

- `to`, `value`, `data`, and `access_list` follow the same semantics as [EIP-1559](./eip-1559.md). `to` MAY be empty to indicate contract creation.
- `value` MUST be `0`. The field is included for RLP compatibility with [EIP-1559](./eip-1559.md), but counterfactual transactions MUST NOT carry an ETH value transfer from the sender. The sender has no public ETH; ETH movement occurs only through application execution and, if applicable, the protocol's post-execution gas-budget refund.
- `fee_nullifier_hash` is a 32-byte value that uniquely identifies the fee note used to pay gas.
- `fee_root` is a 32-byte value that identifies the canonical fee-note Merkle root against which the proof is verified.
- `fee_denomination` is the full ETH value of the fee note. It MUST be one of `FEE_DENOMINATIONS` and MUST be greater than or equal to `gas_limit * max_fee_per_gas`.
- `gas_refund_recipient` is the address that receives unused gas budget from the fee note after execution as a real public ETH transfer.
- `proof_data` is the serialized zero-knowledge proof for spending the canonical fee note. Its format is fixed by this EIP's canonical proof system.

The `nonce` is the sender's nonce, incremented as normal. The sender is recovered from the signature `(y_parity, r, s)` using standard ECDSA recovery over the signing hash:

```
keccak256(COUNTERFACTUAL_TX_TYPE || rlp([chain_id, nonce, max_priority_fee_per_gas,
          max_fee_per_gas, gas_limit, to, value, data, access_list,
          fee_nullifier_hash, fee_root, fee_denomination, gas_refund_recipient, proof_data]))
```

The [EIP-2718](./eip-2718.md) `ReceiptPayload` for this transaction is `rlp([status, cumulative_transaction_gas_used, logs_bloom, logs])`.

### Canonical Fee Note Predeploy

At the block where this EIP activates, the code at `FEE_NOTE_ADDRESS` (`0x14`) MUST be set to the runtime bytecode specified in the [Reference Implementation](#fee-note-contract). `0x14` is the next address after the Contract Payer Transaction payer registry at `0x13`.[^contract-payer-tx]

The fee-note predeploy serves three functions:

1. **Deposit ETH fee notes**: users deposit ETH commitments into a canonical fee-note tree.
2. **Protocol reads**: during validation, the protocol reads nullifier status and recognized Merkle roots from the predeploy.
3. **Protocol writes**: during execution, the protocol marks fee nullifiers as spent in the predeploy's state.

Any application MAY be targeted by a counterfactual transaction. The protocol is only aware of the canonical ETH fee-note system. Application-specific private state transitions, such as privacy-pool withdrawals, remain ordinary EVM execution and are not protocol-registered.

#### Storage Layout

The fee-note predeploy uses fixed storage regions:

```
CURRENT_ROOT_SLOT = 0
NEXT_LEAF_INDEX_SLOT = 1
NULLIFIER_SLOT_BASE = 2
ROOT_HISTORY_SLOT_BASE = 3
ROOT_HISTORY_SIZE = 1024
```

The spent flag for a given `fee_nullifier_hash` is located at:

```
storage[keccak256(abi.encode(NULLIFIER_SLOT_BASE, fee_nullifier_hash))]
```

A nullifier slot value of zero means unspent; any non-zero value means spent.

Recognized Merkle roots are stored in a fixed ring buffer. The history slot for history index `i`, where `0 <= i < ROOT_HISTORY_SIZE`, is:

```
storage[ROOT_HISTORY_SLOT_BASE + i]
```

Let `leaf_index_before` be the value of `storage[NEXT_LEAF_INDEX_SLOT]` immediately before a successful deposit appends a new commitment. After recomputing the new Merkle root `new_root`, the predeploy MUST:

1. Set `storage[CURRENT_ROOT_SLOT] = new_root`.
2. Set `storage[ROOT_HISTORY_SLOT_BASE + (leaf_index_before % ROOT_HISTORY_SIZE)] = new_root`.
3. Set `storage[NEXT_LEAF_INDEX_SLOT] = leaf_index_before + 1`.

At activation, `storage[CURRENT_ROOT_SLOT]` and `storage[ROOT_HISTORY_SLOT_BASE]` MUST be initialized to the canonical empty-tree root derived from this EIP's canonical depth-32 Poseidon Merkle tree. All other root-history slots MUST be zero until overwritten by deposits.

A root is recognized if and only if either:

- `root == storage[CURRENT_ROOT_SLOT]`, or
- there exists some `i` with `0 <= i < ROOT_HISTORY_SIZE` such that `root == storage[ROOT_HISTORY_SLOT_BASE + i]`.

No alternative root-history storage layout or lookup algorithm is permitted.

#### Contract Interface

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.28;

/// @title Fee Notes - EIP-8224
/// @notice Canonical system contract for ETH-denominated fee notes.
///         Users deposit ETH commitments here; the protocol validates
///         and spends fee notes for counterfactual transactions.
contract FeeNotes {
    event Deposited(bytes32 indexed commitment, uint256 amount);

    /// @notice Deposit ETH into the canonical fee-note tree.
    /// @param denominationIndex Index into `FEE_DENOMINATIONS`.
    /// @param commitment The fee-note commitment appended to the tree.
    function deposit(uint8 denominationIndex, bytes32 commitment) external payable;

    /// @notice Returns true if `root` is in the bounded recognized history.
    function isKnownRoot(bytes32 root) external view returns (bool);

    /// @notice Returns true if `nullifierHash` has been spent.
    function isSpent(bytes32 nullifierHash) external view returns (bool);

    /// @notice Returns the current Merkle root.
    function currentRoot() external view returns (bytes32);

    /// @notice Returns the total number of deposited notes for a denomination.
    ///         This is intended for wallet and frontend privacy warnings.
    function denominationDepositCount(uint8 denominationIndex) external view returns (uint256);
}
```

Deposits MUST use fixed denominations. `deposit(denominationIndex, commitment)` MUST revert unless `msg.value == FEE_DENOMINATIONS[denominationIndex]`. The predeploy MUST append commitments sequentially to a single Merkle tree, update `NEXT_LEAF_INDEX_SLOT`, recompute the current root, and write that root into the exact ring-buffer history defined above. `deposit` MUST revert if `NEXT_LEAF_INDEX_SLOT >= 2^32` (tree full). This prevents underfunded commitments and makes note denomination a canonical property of the deposited note. Implementations SHOULD expose enough information for wallets and frontends to estimate denomination-specific anonymity-set size, for example via `denominationDepositCount`.

### ZK Proof Verification

#### Proof System

This draft uses Groth16 over the BN254 (alt_bn128) curve as its current canonical proof system. Verification uses the same elliptic curve arithmetic as [EIP-196](./eip-196.md) (point addition and scalar multiplication) and [EIP-197](./eip-197.md) (pairing check). This bounds verification to a fixed number of curve operations regardless of application complexity.

Future revisions of this EIP may replace the canonical proof system before finalization. After activation, any such change would require a hard fork (see [Post-Quantum Considerations](#post-quantum-considerations)).

Let `BN254_SCALAR_MODULUS = 21888242871839275222246405745257275088548364400416034343698204186575808495617`.

The canonical fee-note circuit is constrained as follows:

- The Merkle tree depth is `32`, allowing `2^32` fee-note leaves.
- Commitments and Merkle internal nodes use Poseidon over the BN254 scalar field.
- A fee note is represented by `(denomination_index, nullifier_secret, trapdoor)`.
- The note commitment is `Poseidon(denomination_index, nullifier_secret, trapdoor)`.
- The nullifier hash is `Poseidon(nullifier_secret)`.
- `denomination_index` selects a value from `FEE_DENOMINATIONS`.
- The circuit embeds the `FEE_DENOMINATIONS` table as constants. It constrains `fee_denomination == FEE_DENOMINATIONS[denomination_index]`, proving the public input matches the note's committed denomination.
- The circuit proves a correct Merkle path from the note commitment to `fee_root`.
- The exact Poseidon instance used by this EIP, including state width, round counts, MDS matrix, and round constants, is consensus-critical. All implementations MUST use the single canonical Poseidon parameterization published in the reference implementation for this EIP.

Equivalent algebraic formulations are not permitted: all implementations MUST use this exact commitment, nullifier, and Merkle-hash scheme.

#### Public Input Encoding

All Groth16 public inputs are single BN254 scalar-field elements. Clients MUST construct them as follows:

- `fee_nullifier_hash` and `fee_root` are encoded as 32-byte big-endian integers and MUST be strictly less than `BN254_SCALAR_MODULUS`; otherwise the transaction is invalid.
- `fee_denomination` and `gas_commitment` are encoded as unsigned big-endian integers and MUST be strictly less than `BN254_SCALAR_MODULUS`; otherwise the transaction is invalid.
- `gas_refund_recipient` and `sender` are encoded as their `uint160` address values lifted into a field element.

`gas_commitment` is computed as the exact integer product `gas_limit * max_fee_per_gas`. If this multiplication overflows `uint256` or yields a value greater than or equal to `BN254_SCALAR_MODULUS`, the transaction is invalid.

#### Public Inputs

The ZK proof MUST commit to the following public inputs:

| Index | Public Input | Description |
| --- | --- | --- |
| 0 | `fee_nullifier_hash` | The fee nullifier (matches `tx.fee_nullifier_hash`) |
| 1 | `fee_root` | A recent Merkle root of the canonical fee-note tree (matches `tx.fee_root`) |
| 2 | `fee_denomination` | The full fee-note denomination (matches `tx.fee_denomination`) |
| 3 | `gas_commitment` | Equal to `gas_limit * max_fee_per_gas` from the transaction |
| 4 | `gas_refund_recipient` | The address receiving unused gas budget |
| 5 | `sender` | The recovered transaction sender |

#### Proof Statement

The proof demonstrates, without revealing the note amount or depositor identity:

1. The prover knows a valid fee note `F` in the canonical fee-note tree at `fee_root`.
2. `fee_nullifier_hash` is correctly derived from `F`.
3. `fee_denomination` is correctly derived from `F` and is one of `FEE_DENOMINATIONS`.
4. `fee_denomination >= gas_commitment`.
5. `gas_refund_recipient` is correctly bound into the proof statement so unused gas budget cannot be redirected by a third party.
6. `sender` is correctly bound into the proof statement so a fee proof cannot be front-run by a different sender.

The key property is not application-specific solvency. It is that the transaction carries an independently spendable ETH fee authorization whose minimum value is known to the protocol before execution begins.

#### Verification Procedure

During validation, the protocol:

1. Reads the canonical fee-note verification key defined by this EIP.
2. Constructs the public input vector from the transaction fields and recovered sender.
3. Performs Groth16 verification using BN254 pairing arithmetic.

`gas_commitment` is not a transaction field. The verifier computes it as `gas_limit * max_fee_per_gas` when constructing the public input vector.

This is bounded cryptographic computation with no EVM code execution and no state-dependent branching.

### Transaction Validation

A counterfactual transaction is valid if and only if all of the following hold:

1. The signature is valid and the sender is recoverable.
2. `chain_id` matches the current chain.
3. `max_fee_per_gas >= base_fee_per_gas`.
4. `max_fee_per_gas >= max_priority_fee_per_gas`.
5. `gas_limit >= intrinsic_gas`. The intrinsic gas for a counterfactual transaction is computed identically to [EIP-1559](./eip-1559.md) (21000 base + calldata costs + access list costs). The `proof_data` field is included in the RLP-encoded transaction payload and therefore contributes to calldata gas costs at the standard per-byte rate.
6. `nonce == sender.nonce`.
7. `value == 0`.
8. `fee_nullifier_hash < BN254_SCALAR_MODULUS`.
9. `fee_root < BN254_SCALAR_MODULUS`.
10. `fee_denomination` is one of `FEE_DENOMINATIONS`.
11. `gas_commitment = gas_limit * max_fee_per_gas` does not overflow `uint256` and satisfies `gas_commitment < BN254_SCALAR_MODULUS`.
12. `fee_denomination >= gas_commitment`.
13. The fee nullifier is unspent: `fee_note_storage[nullifier_slot(fee_nullifier_hash)] == 0`.
14. `FEE_NOTE_ADDRESS.balance >= fee_denomination`.
15. The ZK proof verifies against the canonical fee-note verification key with the public inputs derived from the transaction fields and recovered sender.
16. The `fee_root` public input is recognized by the fee-note predeploy, meaning it is the current root or one of the `ROOT_HISTORY_SIZE` most recent roots.

If any check fails, the transaction is **invalid** and MUST NOT be included in a block.

Implementations SHOULD perform validation checks in ascending cost order and SHOULD run the ZK proof verification in check 15 last, after all cheaper format, range, nonce, denomination, nullifier, balance, and root-recognition checks have succeeded.

The protocol does not require the sender or any application-specific recipient to hold ETH before execution. Gas safety comes from the fee note proved in check 15, not from pre-funded public balances. The fee note is intentionally not limited to fee settlement only: any unused portion may emerge as public ETH at `gas_refund_recipient`.

### Mempool Rules

Clients MUST NOT hold more than one counterfactual transaction per `fee_nullifier_hash` in the transaction pool. If a new counterfactual transaction arrives with a `fee_nullifier_hash` that already has a pending transaction, the client SHOULD replace the existing transaction only if the new transaction has strictly higher `max_fee_per_gas` and `max_priority_fee_per_gas`.

Clients MUST NOT hold more than one counterfactual transaction per recovered sender address in the transaction pool, consistent with the Contract Payer Transaction constraint on sponsored transactions.[^contract-payer-tx]

Clients MUST revalidate counterfactual transactions when:

- The fee-note storage slot for the transaction's `fee_nullifier_hash` changes.
- The fee-note predeploy's balance changes such that it may fall below one or more pending transactions' `fee_denomination`.
- The fee-note system's accepted Merkle root set changes in a way that may invalidate the transaction's `fee_root`.
- The sender's nonce changes.

Fee-nullifier changes are O(1): each fee nullifier maps to at most one pending transaction. Balance changes are not O(1), because they can invalidate multiple pending transactions that share the same denomination bucket. Since `fee_denomination` is drawn from the small fixed set `FEE_DENOMINATIONS`, clients can bound this churn by tracking pending transactions per denomination bucket rather than by arbitrary per-transaction thresholds.

### Transaction Execution

Execution of a valid counterfactual transaction proceeds as follows:

1. **Compute effective gas price**: Let `gas_commitment = gas_limit * max_fee_per_gas` and `effective_gas_price = base_fee_per_gas + min(max_priority_fee_per_gas, max_fee_per_gas - base_fee_per_gas)`.
2. **Consume fee note**: The protocol writes a non-zero value to the fee-note predeploy's nullifier slot for `fee_nullifier_hash`, using the same slot computation as validation check 13. This state change is part of transaction-level fee settlement and is NOT reverted if the EVM call later reverts.
3. **Fee-note release**: Transfer `fee_denomination` from `FEE_NOTE_ADDRESS.balance` to `sender.balance`. This releases the fee note's full value.
4. **Fee escrow**: Deduct `fee_denomination` from `sender.balance`. This locks the full fee-note value so the fee-note funds are never visible to the sender during execution.
5. **Nonce increment**: Set `sender.nonce = sender.nonce + 1`.
6. **Execute**: Execute the transaction body using ordinary [EIP-1559](./eip-1559.md) semantics. The call frame's `msg.value` is `0` regardless of the sender's actual balance. If `to` is non-empty, perform a message call from `sender` to `to` with `data` and `gas_limit - intrinsic_gas` available gas. If `to` is empty, create a contract using the transaction's initcode.
7. **Fee-note refund**: Add `fee_denomination - gas_used * effective_gas_price` to `sender.balance`. `gas_used` here is the final gas consumed after applying the [EIP-3529](./eip-3529.md) gas refund (capped at `gas_used_before_refund / 5`). This returns both unused gas budget and any fee-note value above `gas_commitment`.
8. **Refund return**: Transfer `fee_denomination - gas_used * effective_gas_price` from `sender.balance` to `gas_refund_recipient.balance`. This sends the unused portion of the fee note to the designated recipient.
9. **Priority fee**: Add `gas_used * (effective_gas_price - base_fee_per_gas)` to the coinbase balance.
10. **Base fee burn**: `gas_used * base_fee_per_gas` is burned.

Steps 3-4 and 7-8 are each atomic: the sender's balance is transiently increased (step 3) then immediately decreased (step 4), so the fee-note funds are never visible to the sender during execution. (If the sender already holds ETH from other sources, that pre-existing balance is unaffected and remains available during execution.) Similarly, the refund (step 7) is immediately forwarded (step 8). The net effect is: `fee_denomination` is deducted from the canonical fee-note pool, `gas_used * effective_gas_price` goes to gas settlement (priority fee and burn), and the remainder goes to `gas_refund_recipient`. All intermediate balance movements are real and emit [EIP-7708](./eip-7708.md) Transfer logs.

**Critical difference from the Contract Payer Transaction[^contract-payer-tx]**: In sponsored transactions, gas funding uses `effective_gas_price` and unused gas is returned to the payer. Here, the protocol releases the full `fee_denomination`, uses `gas_used * effective_gas_price` for settlement, and sends the remainder to `gas_refund_recipient`, not back to the canonical fee-note pool. The fee note is fully consumed; the canonical pool does not retain the spread.

**Execution failure semantics**: If the call or creation in step 6 reverts, the application effects are reverted as usual, but steps 2-4 (fee-note consumption and fee escrow) are NOT reverted and gas is still charged. This matches the normal Ethereum rule that the gas payer remains liable even when execution fails. The canonical fee-note pool loses `fee_denomination` regardless of call outcome; the refund recipient receives the unused portion.

### Interaction with Contract Payer Transaction

If a counterfactual transaction bootstraps a smart contract account:

1. The recipient smart contract account holds the application-delivered asset(s) and, if chosen as `gas_refund_recipient`, any unused ETH from the fee note.
2. The recipient calls `authorize` on the Contract Payer Transaction payer registry at `0x13`, registering the sender EOA as its authorized controller.[^contract-payer-tx]
3. All subsequent transactions from the EOA to the smart contract account use Contract Payer Transactions (`0x07`) with standard pre-execution gas escrow and registry validation.[^contract-payer-tx]

In that bootstrap case, the counterfactual transaction is needed only once. After the account is funded and registered in the payer registry, the full Contract Payer Transaction lifecycle takes over.[^contract-payer-tx] Counterfactual transactions remain available for any later call that wants protocol-native shielded gas funding instead of contract-payer sponsorship.

### Interaction with EIP-7708

[EIP-7708](./eip-7708.md) requires that all ETH transfers emit logs. Every balance transfer in the [Transaction Execution](#transaction-execution) steps is a real state change. The two new transfers introduced by this EIP - the fee-note release (step 3, `FEE_NOTE_ADDRESS` -> `sender`) and the refund return (step 8, `sender` -> `gas_refund_recipient`) - MUST each emit exactly one [EIP-7708](./eip-7708.md) Transfer log:

| Field | Fee-note release log (step 3) | Refund return log (step 8) |
| --- | --- | --- |
| `address` | `SYSTEM_ADDRESS` | `SYSTEM_ADDRESS` |
| `topics[0]` | `keccak256('Transfer(address,address,uint256)')` | `keccak256('Transfer(address,address,uint256)')` |
| `topics[1]` | `FEE_NOTE_ADDRESS` | `sender` |
| `topics[2]` | `sender` | `gas_refund_recipient` |
| `data` | `fee_denomination` | `fee_denomination - gas_used * effective_gas_price` |

Where `SYSTEM_ADDRESS` is `0xfffffffffffffffffffffffffffffffffffffffe` as defined by [EIP-4788](./eip-4788.md).

The fee-note release log is emitted before any logs produced by the transaction call, and the refund return log is emitted after all execution logs. This ordering is consensus-critical. All other logs produced by the transaction body are emitted by the standard [EIP-1559](./eip-1559.md) and [EIP-7708](./eip-7708.md) machinery.

### Interaction with EIP-7702

If the sender has an [EIP-7702](./eip-7702.md) delegation designator, the delegation is resolved as normal for code execution purposes. The ZK proof verification and fee-note-based gas authorization are independent of the sender's delegation status.

## Rationale

### What is the architectural prior art?

The fee-note construction follows the established commitment-tree plus nullifier pattern used by earlier privacy systems such as Tornado Cash,[^tornado] Semaphore,[^semaphore] and Zcash.[^zcash] This EIP applies that pattern narrowly to gas funding and limited bootstrap liquidity at the protocol layer, rather than trying to enshrine a general-purpose private transfer system.

That narrow scope is intentional. The protocol validates only the canonical ETH fee note and its gas-funding semantics. Application-specific private state transitions remain ordinary contract execution. This keeps the consensus surface smaller than broader protocol-native privacy-transfer proposals while still addressing a concrete and otherwise unresolved bootstrap problem.

### Why a dedicated fee note instead of charging the recipient or post-state application balance?

The public mempool needs a pre-execution answer to "who can definitely pay for worst-case gas?" A proof that some application recipient will have ETH after execution is not sufficient, because gas must still be payable if the call reverts. A dedicated fee note gives the protocol a static, pre-execution gas authorization that does not rely on post-state balances.

This also reduces privacy risk relative to sending ETH to the controlling EOA just to fund gas. The fee note is simply another spend from the fee-note anonymity set. It authorizes gas without requiring any prior public funding transaction to the EOA.

### Why canonical protocol-native shielded gas funding instead of per-application registration?

The fee lane should be uniform even when the application logic is not. By making fee authorization canonical and protocol-native, this EIP avoids requiring each privacy pool or application contract to register proof systems, nullifier layouts, or validation metadata with the protocol.

That separation is the main composability win: the protocol validates only the ETH fee note, while the transaction body can perform any ordinary call or contract creation. A privacy pool withdrawal becomes just one application of the transaction type rather than the thing the protocol is directly validating.

### Why allow the fee note to emerge as public ETH?

This EIP intentionally allows the unused portion of the fee note to emerge as public ETH at `gas_refund_recipient`. That makes the fee note more than a pure gas-payment artifact: it is a bounded ETH exit path whose maximum public value is the note's public `fee_denomination`.

This is pragmatic for bootstrapping. If the refund recipient is the new smart account, the same transaction both performs the private application call and leaves the account with some public ETH for future operation. It also enables third-party privacy pools of other assets: the application logic can release asset `X`, while the fee note supplies the ETH needed to bootstrap the account into the Contract Payer Transaction sponsorship model.[^contract-payer-tx]

This path is intentionally ETH-only. It does not provide a generic protocol withdrawal mechanism for arbitrary assets; those remain the responsibility of the called application's own execution logic. The design goal is limited bootstrap liquidity for subsequent operation, not a high-capacity withdrawal facility.

The larger denominations, especially `1 ETH` and `5 ETH`, primarily serve this bootstrap role rather than worst-case gas coverage alone. Because `fee_denomination` is public, the amount of ETH emerging through the refund path is transparent even though the fee-note source remains shielded. The capped `5 ETH` maximum reflects that emphasis: the canonical fee-note pool is meant to support privacy-preserving gas funding and limited bootstrap liquidity, not to maximize standalone ETH withdrawal capacity.

This also distinguishes the design from broader privacy-transfer proposals such as [EIP-7503](./eip-7503.md), which aim at protocol-native private ETH transfer semantics rather than narrowly scoped mempool-safe gas funding.

### Why ZK proofs instead of payer registry authorization?

The Contract Payer Transaction[^contract-payer-tx] uses the payer registry for authorization: `registry[tx.to] == sender`. This requires the payer contract to have previously called `authorize(sender)`. That works well for persistent smart accounts, but not for shielded fee sources where revealing the authorization relationship on-chain defeats the privacy goal. A ZK proof provides authorization without revealing which fee-note deposit is being spent. The proof replaces the registry lookup with a cryptographic check that is equally static from the protocol's perspective.

### Why not let the application recipient "pay at the end"?

That would reintroduce the bootstrap problem. The recipient has no ETH before execution, so a public-mempool validator or block builder would have no unconditional guarantee of payment if the call fails. The fee note solves this cleanly by making gas payment independent of whether the application call itself succeeds.

### Why a separate transaction type?

A dedicated [EIP-2718](./eip-2718.md) type allows the sender to explicitly signal shielded fee-note semantics. Without this, the protocol would need heuristics to determine whether a transaction should use ordinary sender-paid gas, Contract Payer Transaction sponsorship,[^contract-payer-tx] or protocol-native shielded gas funding. The separate type also allows mempool rules (per-nullifier deduplication) that are specific to counterfactual transactions.

### Why per-nullifier mempool deduplication?

Each gas authorization has a unique fee nullifier. Once that nullifier is spent, the corresponding proof cannot be replayed. Deduplicating by fee nullifier ensures O(1) revalidation when the relevant state changes.

### Why fund gas at max_fee_per_gas instead of effective_gas_price?

The Contract Payer Transaction[^contract-payer-tx] transfers `gas_limit * effective_gas_price` from the payer, letting the spread between `max_fee_per_gas` and `effective_gas_price` remain with the payer. This makes sense for ongoing sponsorship where the payer is a persistent smart account that wants to minimize gas costs. For a fee note, the situation is different: the note is a one-time deposit being fully consumed. The user chose a fee denomination to cover worst-case gas; any portion not needed for gas should flow to the refund recipient, not remain stranded in the canonical fee-note pool. Releasing the full `fee_denomination` ensures the entire fee note value is accounted for: gas costs go to settlement, and the remainder goes to `gas_refund_recipient`.

### How does the anonymity set bootstrap?

The fee-note system has a real cold-start problem: early depositors have weak privacy because the anonymity set is initially small. This EIP does not solve that at the protocol layer.

Reasonable bootstrap paths include:

- privacy-pool frontends depositing fee notes alongside ordinary pool deposits,
- wallets opportunistically creating fee notes during normal user activity,
- or application-specific onboarding flows that create fee notes ahead of the first counterfactual transaction.

Users SHOULD treat small anonymity sets as weak privacy rather than assuming the canonical fee-note pool is private by default from genesis. Wallets and frontends SHOULD surface denomination-specific anonymity-set warnings when the available crowd is too small for the user's stated privacy goals.

### What is the deposit cost tradeoff?

The fee-note predeploy must update a Poseidon-based Merkle tree on each deposit. Without a dedicated Poseidon precompile, this is expected to be materially more expensive than a simple ETH transfer.

For small denominations such as `0.001 ETH`, deposit gas overhead may be a significant fraction of the note's value. For a depth-32 Poseidon tree, a deposit is expected to be substantially more expensive than a plain ETH transfer and may be economically unattractive on L1 at elevated gas prices. This is a practical tradeoff of protocol-native shielded gas funding. A future hard fork MAY introduce a Poseidon precompile or other optimization if deposit cost proves too high in practice.

Poseidon is used because it is efficient inside zero-knowledge circuits relative to conventional hash functions, even though it is comparatively expensive to compute in today's EVM without a dedicated precompile.[^poseidon]

The denomination ladder is also intentionally conservative at the high end. While a fee note can be used as a bounded shielded ETH exit via the refund path, that is not its primary purpose. The `5 ETH` maximum keeps the largest tier useful for account bootstrap and limited ETH release without turning the canonical fee-note pool into a high-capacity withdrawal rail by default.

### Why Groth16 over BN254?

Groth16 produces constant-size proofs (~192 bytes) with fast verification (~3 pairings on BN254). The BN254 curve operations are already available as Ethereum precompiles ([EIP-196](./eip-196.md), [EIP-197](./eip-197.md)), so no new cryptographic primitives are required. Verification cost is bounded and independent of circuit complexity, making it suitable for mempool validation where predictable cost is essential.

This EIP therefore follows the same basic proving strategy used by several earlier production privacy systems, but applies it to a much narrower protocol task: proving fee-note ownership and authorization for a single transaction.[^groth16]

For this transaction type, proof size and verifier cost matter more than prover throughput. Every full node must validate every counterfactual transaction, and every proof is carried inside a transaction. That makes compact proofs and cheap verification unusually important.

Universal-setup systems such as PLONK and fflonk are plausible alternatives. Their main advantage is that future circuit revisions can reuse the same universal setup rather than requiring a new circuit-specific ceremony. Their tradeoff is somewhat larger proofs and somewhat heavier verification than Groth16. If the fee-note circuit changes materially before finalization, that tradeoff may become more attractive in a future revision of this EIP.

Transparent systems such as STARKs were also considered. Their main advantages are no trusted setup and a stronger long-term post-quantum story. Their main disadvantage for this EIP is proof size: carrying tens of kilobytes of proof data in every counterfactual transaction would impose substantial bandwidth and mempool overhead. Absent new precompiles or a different protocol design, they are not the best initial fit for this transaction type.

Hybrid designs such as STARK-in-SNARK were not chosen. For a small fixed circuit like the fee-note spend, they add substantial prover complexity without preserving the main benefits usually sought from transparent proofs. In particular, the outer SNARK still determines compact verification and still reintroduces trusted-setup or non-post-quantum assumptions depending on the wrapper used.

## Backwards Compatibility

This EIP introduces a new transaction type and a new predeploy contract. It does not modify the behavior of existing transaction types.

The `COUNTERFACTUAL_TX_TYPE` value `0x08` does not conflict with any finalized transaction types: `0x01` ([EIP-2930](./eip-2930.md)), `0x02` ([EIP-1559](./eip-1559.md)), `0x03` ([EIP-4844](./eip-4844.md)), `0x04` ([EIP-7702](./eip-7702.md)), or `0x07` (the Contract Payer Transaction).[^contract-payer-tx]

No existing application contract is required to register with the protocol in order to be targeted by a counterfactual transaction.

## Deployment Considerations

Advancing this EIP to cross-client review requires publication of the canonical fee-note predeploy bytecode, the canonical verification key, the canonical circuit artifacts, the canonical Poseidon parameterization and constants, and executable cross-client test vectors. These are consensus-critical, not optional implementation details.

Until those exact artifacts and cross-client test vectors are published, this EIP remains incomplete and MUST NOT advance beyond draft status.

Client implementations need four non-trivial components:

1. Counterfactual transaction envelope parsing and signing-hash support.
2. Groth16 verification in the transaction validation path.
3. The fee-note predeploy, including fixed-denomination deposit handling and Poseidon-based Merkle tree updates.
4. The protocol storage-write path that marks fee nullifiers as spent during execution.

These items SHOULD be tracked explicitly in devnet planning and cross-client implementation matrices before the EIP is considered ready for mainnet review.

### Example Privacy Pool Integration

A privacy pool can support counterfactual transactions without any protocol registration. It just exposes its normal withdrawal entrypoint:

```solidity
interface IPayerRegistry {
    function authorize(address sender) external;
}

IPayerRegistry constant PAYER_REGISTRY = IPayerRegistry(0x0000000000000000000000000000000000000013);

contract PrivacyPool {
    mapping(bytes32 => bool) public nullifiers;

    /// @notice Process a withdrawal. Called via counterfactual transaction.
    ///         The protocol verifies only the fee-note proof. The contract
    ///         verifies its own withdrawal proof and state transition.
    function withdraw(
        bytes calldata withdrawalProof,
        bytes32 withdrawalNullifierHash,
        bytes32 salt,
        address initialController,
        uint256 amount,
        bytes32 expectedRecipientCodeHash
    ) external {
        require(!nullifiers[withdrawalNullifierHash], "Nullifier already spent");
        require(_verifyWithdrawalProof(withdrawalProof, withdrawalNullifierHash, salt, initialController, amount), "Invalid proof");

        nullifiers[withdrawalNullifierHash] = true;

        bytes memory initCode = abi.encodePacked(type(SmartAccount).creationCode, abi.encode(initialController));
        require(keccak256(initCode) == expectedRecipientCodeHash, "Unexpected initcode");

        address deployed;
        assembly {
            deployed := create2(amount, add(initCode, 0x20), mload(initCode), salt)
        }
        require(deployed != address(0), "Create2 failed");
    }

    function _verifyWithdrawalProof(
        bytes calldata, /* withdrawalProof */
        bytes32, /* withdrawalNullifierHash */
        bytes32, /* salt */
        address, /* initialController */
        uint256 /* amount */
    ) internal view returns (bool) {
        return true; // placeholder
    }

    receive() external payable {}
}
```

### Example Bootstrap Flow

```solidity
/// @notice A smart contract account deployed by the privacy pool
///         during withdrawal. Calls authorize in its constructor
///         to enable Contract Payer Transaction sponsored transactions immediately.
contract SmartAccount {
    address public controller;

    constructor(address initialController) payable {
        controller = initialController;
        // Register the controller for Contract Payer Transaction sponsored transactions.
        // After this, the controller EOA never needs ETH again.
        PAYER_REGISTRY.authorize(initialController);
    }

    function execute(address target, uint256 val, bytes calldata data) external {
        require(msg.sender == controller, "Not controller");
        (bool ok,) = target.call{value: val}(data);
        require(ok, "Execution failed");
    }

    receive() external payable {}
}
```

The full bootstrap flow:

1. Alice deposits 10 ETH into `PrivacyPool`, receiving a withdrawal commitment. Separately, she deposits a `0.01 ETH` fee note into the canonical fee-note contract at `0x14`.
2. Time passes. Alice generates a fresh EOA keypair.
3. Alice constructs a counterfactual transaction:
   - `to` = `PrivacyPool` address
   - `data` = `withdraw(withdrawalProof, withdrawalNullifierHash, salt, freshEOA, 10 ether, keccak256(initCode))`
   - `recipient` is the CREATE2 address derived from `salt`, `PrivacyPool`, and `initCode = SmartAccount.creationCode ++ abi.encode(freshEOA)`
   - `proof_data` = Groth16 proof for spending the canonical fee note
   - `fee_nullifier_hash` = her fee note's nullifier
   - `fee_root` = a recent recognized root of the canonical fee-note tree
   - `fee_denomination` = `0.01 ether`
   - `gas_refund_recipient` = `recipient` (unused gas budget goes to the new smart account)
   - Signed by the fresh EOA
4. The mempool validates: fee-note proof OK, fee nullifier unspent, fee root recognized, `0.01 ether >= gas_limit * max_fee_per_gas`, and the fee-note predeploy balance is sufficient.
5. A block builder includes the transaction.
6. The protocol consumes the fee nullifier, releases `0.01 ether` from the canonical fee-note pool into fee escrow, and executes the call.
7. `PrivacyPool.withdraw` verifies the withdrawal proof, marks the withdrawal nullifier as spent, deploys `SmartAccount` via `CREATE2` with `10 ETH`, and runs the payable constructor. The `SmartAccount` constructor calls `PAYER_REGISTRY.authorize(freshEOA)`.
8. After execution, gas is settled: `gas_used * effective_gas_price` goes to the priority fee and burn, and the remainder of the fee note (`0.01 ether - gas_used * effective_gas_price`) is sent to `gas_refund_recipient`.
9. Result: `SmartAccount` holds the withdrawn ETH plus the unused portion of the fee note. The fresh EOA is authorized in the payer registry. All future transactions use Contract Payer Transaction type `0x07`.[^contract-payer-tx]

The same structure works for third-party privacy pools of other assets. In that case, step 7 would transfer asset `X` to `SmartAccount`, while the fee note still provides the ETH bootstrap via step 8.

## Security Considerations

### Mempool Denial of Service

ZK proof verification is computationally more expensive than the storage reads required by the Contract Payer Transaction.[^contract-payer-tx] A Groth16 verification over BN254 requires approximately 3 pairing operations and several point multiplications. While bounded and predictable, this is roughly 100-1000x more expensive than a storage read in CPU time.

Implementations SHOULD perform all cheaper prechecks before running proof verification, and SHOULD rate-limit counterfactual transaction submissions using per-peer or equivalent admission throttling so proof verification cannot consume excessive node resources. The per-sender and per-nullifier mempool caps provide additional natural bounds, but they do not eliminate the need for front-door throttling of unique invalid proofs.

### Nullifier Frontrunning

Between mempool acceptance and block inclusion, the fee nullifier could be spent by a different transaction. This would invalidate the counterfactual transaction. Block builders MUST re-check fee-nullifier state at inclusion time. The risk is bounded: each fee nullifier can be spent exactly once, so at most one counterfactual transaction per fee nullifier can be invalidated.

### Fee Authorization Consumption

The fee nullifier is consumed in step 2 of [Transaction Execution](#transaction-execution), outside normal EVM rollback semantics, so that gas remains payable even if the call in step 6 reverts. This is intentional, but it means users MUST treat the fee note as irrevocably spent once the transaction is included, regardless of whether the application call itself succeeds. The unused portion of the fee note is still refunded to `gas_refund_recipient`, so the user's loss is bounded to `gas_used * effective_gas_price`, not the full `fee_denomination`.

### Proof Malleability

The ZK proof binds to the transaction's public inputs (`fee_nullifier_hash`, `fee_root`, `fee_denomination`, `gas_commitment`, `gas_refund_recipient`, and `sender`). If any of those values changes, the proof becomes invalid. Implementations MUST verify that the public inputs derived from the transaction and recovered sender match those committed to in the proof, to prevent an attacker from reusing a valid proof with modified transaction parameters.

### Privacy Leakage via Gas

The gas used by a counterfactual transaction is publicly visible. If different private actions lead to meaningfully different gas consumption, gas usage could leak information about the underlying application state transition. Applications that care about privacy SHOULD keep gas cost as independent as possible from private amounts and branches.

### Canonical Circuit Trust

The canonical fee-note verification key determines what fee proofs the protocol accepts. If the circuit or verification key is wrong, invalid fee-note spends could be accepted or valid fee-note spends rejected. The fee-note circuit and verification key are therefore consensus-critical artifacts and MUST be audited and published canonically as part of this EIP.

Because the current draft proof system is Groth16, the fee-note system also depends on a circuit-specific trusted setup ceremony. If the ceremony's toxic waste is compromised, an attacker may be able to forge proofs and extract ETH from the fee-note predeploy without valid deposits. The setup transcript, ceremony procedure, and verification method are therefore security-critical parts of the deployment process.

Any change to the fee-note circuit under Groth16 requires a new trusted setup and a new verification key. This is one reason the circuit and Poseidon parameterization must be stabilized before the EIP can progress beyond draft review.

### Privacy Leakage via Fee Notes

A separate fee note introduces another spent nullifier that can be correlated with the transaction. This is preferable to publicly funding the controlling EOA, but it is still an additional observable artifact. Deposit-side privacy is not provided by the canonical fee-note predeploy itself: `deposit(...)` is a normal public transaction that reveals the depositor address and denomination tier. The privacy property is that later spending reveals only the nullifier and public transaction bindings, not the link back to a particular deposit. In this design, the fee note is also an intentional bounded public ETH exit path via `gas_refund_recipient`, so users and applications should treat it as part of the overall privacy surface rather than as invisible fee plumbing. The canonical fee-note pool SHOULD maintain a large anonymity set, and users SHOULD avoid deterministic fee-note selection patterns.

### Protocol-Level Storage Writes

Step 2 of [Transaction Execution](#transaction-execution) requires the protocol to write directly to the canonical fee-note predeploy's nullifier storage. This is much closer to existing protocol writes ([EIP-4788](./eip-4788.md), [EIP-2935](./eip-2935.md)) than the previous per-application design, but it is still consensus-critical. Implementations must ensure this write path is bounded (one storage slot, deterministic location, non-zero value) and cannot be abused to corrupt unrelated predeploy state.

### Interaction with Base Fee Volatility

Because the fee note commits to a public `fee_denomination`, while actual gas is charged at `gas_used * effective_gas_price`, the consumed fee authorization is conservative by construction. Unused fee-note value is refunded to `gas_refund_recipient`.

### Balance-Churn Revalidation

Changes to `FEE_NOTE_ADDRESS.balance` can invalidate pending counterfactual transactions if the remaining balance falls below their `fee_denomination`. See [Mempool Rules](#mempool-rules) for the denomination-bucket tracking approach that bounds this churn.

### Post-Quantum Considerations

BN254 and Groth16 are not post-quantum secure. A sufficiently capable quantum adversary could break the underlying discrete-log assumptions and forge proofs. This is not an immediate practical risk, but it is a real long-term limitation.

Before finalization, future revisions of this EIP may replace the current draft proof system with a different canonical proof system. After activation, the proof system, verification key, and fee-note predeploy could be replaced only by a future hard fork.

Such a hard fork can preserve existing fee notes only if it keeps the same note commitment, nullifier, and Merkle semantics, or defines an explicit migration path for legacy notes. Changing the verifier alone is compatible with existing notes; changing the note model is not automatically compatible.
