## Abstract

This EIP introduces a new [EIP-2718](./eip-2718.md) typed transaction (`TransactionType = 0x05`) called SchemedTransaction. A SchemedTransaction allows the sender to select a signature algorithm via an explicit `scheme_id` byte in the transaction envelope. Three schemes are defined at launch: secp256k1 (ECDSA), secp256r1 (P256), and Falcon-512 (post-quantum).

The sender address is derived deterministically by recovering (or verifying) the public key from the signature and hashing it. For `scheme_id = 0x00` (secp256k1), address derivation is exactly the same as legacy Ethereum. For other schemes, the `scheme_id` is prefixed before hashing to provide address domain separation.

Because post-quantum signatures are significantly larger than ECDSA signatures, Falcon transactions incur an additional gas surcharge proportional to the signature and public key size.

## Motivation

### The post-quantum imperative

Quantum computers will eventually break ECDSA over secp256k1. The Ethereum Foundation has declared post-quantum security a top protocol priority, but no concrete, user-facing transaction type currently lets EOAs opt in to post-quantum signing.

Many users and wallets need a simpler primitive: sign a standard transaction with a different algorithm, have the protocol verify it natively, and derive the sender address the same way it always has — from the public key embedded in the signature. SchemedTransaction provides exactly this and nothing more.

### Passkey / WebAuthn support

[EIP-7951](./eip-7951.md) defines a P256 precompile, but wallets using WebAuthn or Secure Enclave keys still cannot natively sign Ethereum transactions. SchemedTransaction extends native support to P256, enabling passkey-based wallets without smart contract wrappers.

### Minimal protocol surface

Rather than introducing new opcodes, account abstractions, or execution frame models, SchemedTransaction is a single new transaction type that reuses the existing [EIP-1559](./eip-1559.md) fields with only two changes: the legacy `v, r, s` fields are replaced by `scheme_id` and `signature_data`.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) and [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174).

### Constants

| Name | Value | Description |
|------|-------|-------------|
| `SCHEMED_TX_TYPE` | `0x05` | [EIP-2718](./eip-2718.md) transaction type byte |
| `SCHEME_SECP256K1` | `0x00` | secp256k1 ECDSA |
| `SCHEME_P256` | `0x01` | secp256r1 / P256 (per [EIP-7951](./eip-7951.md)) |
| `SCHEME_FALCON` | `0x02` | Falcon-512 |
| `FALCON_SIG_MAX_LEN` | `666` | Maximum compressed Falcon-512 signature length (bytes) |
| `FALCON_PK_LEN` | `897` | Falcon-512 public key length (bytes) |

The `scheme_id` space ranges from `0x00` to `0xFF`, allowing up to 256 registered algorithms. New schemes MUST be specified via a distinct EIP.

### Transaction Envelope

A SchemedTransaction is serialized as:

```
0x05 || rlp([
    chain_id,
    nonce,
    max_priority_fee_per_gas,
    max_fee_per_gas,
    gas_limit,
    to,
    value,
    data,
    access_list,
    scheme_id,
    signature_data
])
```

#### Field Definitions

| Field | Type | Description |
|-------|------|-------------|
| `chain_id` | `uint256` | [EIP-155](./eip-155.md) chain identifier |
| `nonce` | `uint64` | Sender account nonce |
| `max_priority_fee_per_gas` | `uint256` | [EIP-1559](./eip-1559.md) priority fee |
| `max_fee_per_gas` | `uint256` | [EIP-1559](./eip-1559.md) max fee |
| `gas_limit` | `uint64` | Gas limit |
| `to` | `Address \| null` | Recipient address, or null for contract creation |
| `value` | `uint256` | Wei to transfer |
| `data` | `bytes` | Calldata |
| `access_list` | `List[AccessListEntry]` | [EIP-2930](./eip-2930.md) access list |
| `scheme_id` | `uint8` | Signature algorithm identifier (`0x00`–`0xFF`) |
| `signature_data` | `bytes` | Algorithm-specific signature bytes (see [Signature Formats](#signature-formats)) |

This is intentionally identical to an [EIP-1559](./eip-1559.md) (type 2) transaction with two changes: the legacy `v, r, s` fields are replaced by `scheme_id` and `signature_data`.

### Signing Data

The sender signs the following hash:

```
signing_hash = keccak256(SCHEMED_TX_TYPE || rlp([
    chain_id,
    nonce,
    max_priority_fee_per_gas,
    max_fee_per_gas,
    gas_limit,
    to,
    value,
    data,
    access_list,
    scheme_id
]))
```

Note: `signature_data` is excluded from the signing hash.

### Signature Formats

#### SCHEME_SECP256K1 (`0x00`)

`signature_data = rlp([y_parity, r, s])`

- `y_parity`: `uint8` — Recovery ID (0 or 1).
- `r`: `uint256` — ECDSA signature component.
- `s`: `uint256` — ECDSA signature component. MUST satisfy `s <= secp256k1n / 2` per [EIP-2](./eip-2.md).

The public key is recovered via `ecrecover(signing_hash, y_parity, r, s)`.

The sender address is derived as:

```
sender = keccak256(recovered_public_key)[12:]
```

This is identical to legacy Ethereum address derivation.

#### SCHEME_P256 (`0x01`)

`signature_data = rlp([public_key, r, s])`

- `public_key`: `bytes` — 64-byte uncompressed P256 public key (without the `0x04` prefix).
- `r`: `uint256` — ECDSA signature component.
- `s`: `uint256` — ECDSA signature component. MUST satisfy `s <= p256n / 2`.

The signature is verified by calling the P256 verification algorithm with the `signing_hash`, `(r, s)`, and `public_key`.

The sender address is derived as:

```
sender = keccak256(0x01 || public_key)[12:]
```

The `0x01` prefix provides domain separation so that a P256 key can never collide with a secp256k1 address.

#### SCHEME_FALCON (`0x02`)

`signature_data = rlp([public_key, signature])`

- `public_key`: `bytes` — 897-byte Falcon-512 public key.
- `signature`: `bytes` — Variable-length compressed Falcon-512 signature (MUST NOT exceed 666 bytes).

The signature is verified using the Falcon-512 verification algorithm with the `signing_hash`, `signature`, and `public_key`.

The sender address is derived as:

```
sender = keccak256(0x02 || public_key)[12:]
```

### Intrinsic Gas

The intrinsic gas for a SchemedTransaction is calculated as:

```
intrinsic_gas = 21000
    + calldata_gas(data)
    + access_list_gas(access_list)
    + signature_gas(scheme_id, signature_data)
```

Where `signature_gas` is defined per scheme:

| Scheme | `signature_gas` | Rationale |
|--------|----------------|-----------|
| `SCHEME_SECP256K1` | 0 | Included in base 21000 (same as today) |
| `SCHEME_P256` | 3450 | P256 verification cost (per [EIP-7951](./eip-7951.md) precompile pricing) |
| `SCHEME_FALCON` | `len(public_key) * 16 + len(signature) * 16 + 10000` | Covers calldata-equivalent cost of the larger key/signature material plus verification overhead |

For Falcon-512 with a 897-byte public key and a 666-byte signature, the worst-case signature gas is: `897 * 16 + 666 * 16 + 10000 = 14352 + 10656 + 10000 = 35008`.

### Transaction Validation

A valid SchemedTransaction MUST satisfy all of the following:

1. `scheme_id` MUST be one of `0x00`, `0x01`, or `0x02`.
2. `signature_data` MUST decode to the format specified for the given `scheme_id`.
3. The signature MUST be valid for the `signing_hash` under the specified scheme.
4. For `SCHEME_SECP256K1`: `s <= secp256k1n / 2` per [EIP-2](./eip-2.md).
5. For `SCHEME_P256`: `s <= p256n / 2`.
6. For `SCHEME_FALCON`: `len(signature) <= 666` and `len(public_key) == 897`.
7. The derived sender address MUST exist as an EOA (not a contract) with nonce equal to the transaction `nonce`.
8. The sender's nonce MUST be incremented by one after execution.
9. The transaction MUST have sufficient gas to cover the intrinsic gas including `signature_gas`.

### Receipt

The [EIP-2718](./eip-2718.md) `ReceiptPayload` for this transaction is:

```
rlp([status, cumulative_transaction_gas_used, logs_bloom, logs])
```

Identical to [EIP-1559](./eip-1559.md) receipts.

## Rationale

### Why a new transaction type instead of account abstraction?

Account abstraction ([EIP-4337](./eip-4337.md), [EIP-7702](./eip-7702.md)) provides maximum flexibility but requires smart contract wallets, bundlers, and a significant ecosystem migration. SchemedTransaction targets the common case — an EOA that wants to use a different signature algorithm — with zero ecosystem overhead. The two approaches are complementary.

### Why `scheme_id` in the transaction envelope?

Including `scheme_id` explicitly (rather than inferring from signature length) makes parsing unambiguous, simplifies client implementations, and allows the same public key format across future schemes without ambiguity.

### Why domain-separated addresses?

Prefixing `scheme_id` before the public key hash for non-secp256k1 schemes prevents address collisions across key types. A P256 key and a secp256k1 key that happen to produce the same raw hash will derive different addresses. The exception is `scheme_id = 0x00`, which preserves exact compatibility with existing Ethereum addresses.

### Why charge extra gas for larger signatures?

Post-quantum signatures are 10–20× larger than ECDSA signatures. Without a gas surcharge, a block full of Falcon transactions would consume significantly more bandwidth and storage than a block of ECDSA transactions at the same gas cost. The surcharge ensures economic parity with calldata pricing.

### Why Falcon-512 specifically?

Falcon-512 (NIST FIPS 206) offers the smallest combined signature + public key size among NIST-standardized lattice-based schemes at the 128-bit security level. CRYSTALS-Dilithium signatures are 2–3× larger. Falcon's compact signatures make it the most practical choice for on-chain transactions where every byte has a gas cost.

## Backwards Compatibility

This EIP introduces a new [EIP-2718](./eip-2718.md) transaction type and does not modify the behavior of existing transaction types. Existing wallets and contracts are unaffected.

Transactions with `scheme_id = 0x00` produce identical sender addresses to legacy (type 0), [EIP-2930](./eip-2930.md) (type 1), and [EIP-1559](./eip-1559.md) (type 2) transactions signed by the same secp256k1 key. This means users can migrate to SchemedTransaction without changing their address.

Users of P256 or Falcon-512 will have new addresses derived from their respective public keys with domain separation.

## Security Considerations

### Address collision across schemes

The `scheme_id` prefix in address derivation for non-secp256k1 schemes prevents cross-scheme collisions. An attacker who finds a P256 key matching a secp256k1 address cannot use it because the derived addresses differ.

### Quantum threat timeline

Falcon-512 provides NIST Level I security (equivalent to AES-128) against quantum adversaries. This is considered sufficient for the medium term. Future EIPs may add Falcon-1024 or other schemes via the `scheme_id` extension mechanism if higher security levels are needed.

### Signature malleability

All three schemes enforce canonical signature forms (low-s for ECDSA variants, compressed format for Falcon) to prevent transaction malleability.

### Key management

Users who generate P256 or Falcon-512 keys are responsible for securing them. Unlike secp256k1, these keys cannot be derived from a standard BIP-32/BIP-39 mnemonic. Wallet developers SHOULD define key derivation standards for these schemes in companion EIPs.

### Denial-of-service via large signatures

The `FALCON_SIG_MAX_LEN` cap of 666 bytes and the gas surcharge together prevent abuse. The gas cost ensures that blocks cannot be filled with oversized signatures without paying proportional fees.
