## Abstract

ERC-8183 defines a standard interface for **agentic job contracts** — escrow agreements between a *client* (human or agent) and a *provider* (AI agent) for the delivery of work. The protocol introduces a five-state job lifecycle (`Open → Funded → Submitted → Completed | Rejected | Expired`), a designated *evaluator* role for attestation, and optional integration with ERC-8004 identity and reputation registries.

ERC-8183 enables permissionless, trust-minimized, and composable commerce for autonomous AI agents operating on EVM-compatible chains.

---

## Motivation

The proliferation of autonomous AI agents capable of executing complex tasks — software development, data analysis, content creation, API orchestration — creates an urgent need for a standard for **agent-to-agent and human-to-agent commerce**.

Existing standards are insufficient:

- **ERC-20 / ERC-721** handle asset transfer but not work agreements.
- **Multisig escrow patterns** are ad hoc and not interoperable.
- **ERC-8004** establishes agent identity and reputation but does not define commerce mechanics.

Without a shared standard:

1. Each platform reinvents escrow logic, fragmenting the agent economy.
2. Clients cannot verify provider reputation at the protocol level.
3. On-chain proof of completed work is absent, preventing composability.
4. Multi-agent pipelines cannot delegate sub-jobs to one another predictably.

ERC-8183 fills this gap by specifying a minimal, composable interface for agent work contracts, with an extensible evaluator role that supports anything from a human-controlled EOA to a zero-knowledge attestation registry.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

### Definitions

| Term | Meaning |
|------|---------|
| **Client** | Address that creates a job and funds payment. MAY be an EOA, multisig, or another smart contract / agent. |
| **Provider** | Address that delivers the work described in the job. SHOULD be an ERC-8004 registered agent. |
| **Evaluator** | Trusted address (or contract) that attests completion or rejection. MAY be the client, an arbitration contract, or an on-chain validation registry. |
| **Budget** | Payment escrowed in the contract. MAY be ETH (token = `address(0)`) or any ERC-20 token. |
| **Deliverable** | Off-chain or on-chain URI/hash set by the provider when submitting work. |

### Job Lifecycle

```
          createJob()
              │
           [Open]
              │ fund() (or inline with budget > 0)
           [Funded]
              │ submit()
          [Submitted]
              │
     ┌────────┴────────┐
   complete()       reject()
  [Completed]     [Rejected]
  
  claimRefund() (after expiredAt, from Funded or Submitted)
  [Expired]
```

**State invariants:**
- `Completed` and `Rejected` are terminal states.
- `Expired` is terminal; only reachable after `block.timestamp >= expiredAt`.
- Budget MUST NOT be double-spent: exactly one terminal transition releases funds.

### Interface

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.20;

/// @title IERC8183 — Agentic Commerce Protocol
/// @notice Standard interface for job contracts between clients and AI agent providers.
/// @dev EIP-8183 DRAFT. Implements EIP-165 with interfaceId = 0x8183_0001 (placeholder; final to be assigned).
interface IERC8183 {

    // ── Enumerations ─────────────────────────────────────────────────────────

    enum Status { Open, Funded, Submitted, Completed, Rejected, Expired }

    // ── Structs ───────────────────────────────────────────────────────────────

    struct Job {
        address client;
        address provider;    // address(0) if not yet assigned
        address evaluator;
        address token;       // address(0) = ETH; ERC-20 address otherwise
        uint256 budget;
        uint256 expiredAt;   // unix timestamp
        Status  status;
        string  description;
        string  deliverable; // set by provider on submit()
        string  reason;      // set by evaluator on complete() or reject()
    }

    // ── Events ────────────────────────────────────────────────────────────────

    /// @notice Emitted when a job is created.
    event JobCreated(
        uint256 indexed jobId,
        address indexed client,
        address evaluator,
        string  description
    );

    /// @notice Emitted when a budget is set or updated (before Funded).
    event BudgetSet(uint256 indexed jobId, uint256 amount, address token);

    /// @notice Emitted when funds are escrowed.
    event JobFunded(uint256 indexed jobId, uint256 amount);

    /// @notice Emitted when a provider is assigned.
    event ProviderSet(uint256 indexed jobId, address indexed provider);

    /// @notice Emitted when provider submits deliverable.
    event WorkSubmitted(
        uint256 indexed jobId,
        address indexed provider,
        string  deliverable
    );

    /// @notice Emitted when evaluator marks job complete and provider is paid.
    event JobCompleted(
        uint256 indexed jobId,
        address indexed provider,
        uint256 payment,
        string  reason
    );

    /// @notice Emitted when evaluator rejects the work and client is refunded.
    event JobRejected(uint256 indexed jobId, string reason);

    /// @notice Emitted when job expires and client claims refund.
    event JobExpired(uint256 indexed jobId);

    // ── Write Functions ────────────────────────────────────────────────────────

    /**
     * @notice Create a new job.
     * @param provider   Agent that will do the work. MAY be address(0) if provider is not yet known.
     * @param evaluator  Address that attests completion/rejection. MUST NOT be address(0).
     * @param token      ERC-20 token address for payment, or address(0) for ETH.
     * @param budget     Payment amount. MAY be 0 if budget will be set later via setBudget().
     * @param expiredAt  Unix timestamp after which the client or anyone MAY claim a refund.
     * @param description Human- or machine-readable job brief.
     * @return jobId     Monotonically increasing job identifier.
     */
    function createJob(
        address provider,
        address evaluator,
        address token,
        uint256 budget,
        uint256 expiredAt,
        string  calldata description
    ) external returns (uint256 jobId);

    /**
     * @notice Set or update budget before funding.
     *         MUST revert if job status is not Open.
     *         MUST revert if caller is not the client.
     */
    function setBudget(uint256 jobId, uint256 amount) external;

    /**
     * @notice Assign provider after job creation.
     *         MUST revert if caller is not the client.
     *         MUST revert if job status is not Open.
     *         MAY be called multiple times while Open (re-assignment).
     */
    function setProvider(uint256 jobId, address provider) external;

    /**
     * @notice Fund the job escrow. Moves Open → Funded.
     *         For ETH payments: msg.value MUST equal job.budget.
     *         For ERC-20 payments: caller MUST have approved this contract for job.budget.
     *         MUST revert if job status is not Open.
     *         MUST revert if job.budget == 0.
     */
    function fund(uint256 jobId) external payable;

    /**
     * @notice Provider submits work. Moves Funded → Submitted.
     *         MUST revert if caller is not job.provider.
     *         MUST revert if job status is not Funded.
     * @param deliverable URI or hash of the delivered work (IPFS CID, HTTPS URL, or on-chain data).
     */
    function submit(uint256 jobId, string calldata deliverable) external;

    /**
     * @notice Evaluator marks work complete. Moves Submitted → Completed.
     *         Releases budget to provider (minus optional platform fee).
     *         MUST revert if caller is not job.evaluator.
     *         MUST revert if job status is not Submitted.
     * @param reason Human- or machine-readable completion attestation.
     */
    function complete(uint256 jobId, string calldata reason) external;

    /**
     * @notice Evaluator rejects work. Moves Funded/Submitted → Rejected.
     *         Refunds budget to client.
     *         MUST revert if caller is not job.evaluator (or job.client if Open).
     * @param reason Human- or machine-readable rejection rationale.
     */
    function reject(uint256 jobId, string calldata reason) external;

    /**
     * @notice Claim refund after expiry. Moves Funded/Submitted → Expired.
     *         Refunds budget to client. Anyone MAY call after expiredAt.
     *         MUST revert if block.timestamp < job.expiredAt.
     *         MUST revert if job status is not Funded or Submitted.
     */
    function claimRefund(uint256 jobId) external;

    // ── Read Functions ────────────────────────────────────────────────────────

    /**
     * @notice Return the full Job struct for a given jobId.
     */
    function getJob(uint256 jobId) external view returns (Job memory);

    /**
     * @notice Return true if block.timestamp >= job.expiredAt.
     */
    function isExpired(uint256 jobId) external view returns (bool);
}
```

### EIP-165 Interface Detection

Contracts implementing ERC-8183 MUST implement EIP-165. The `supportsInterface` function MUST return `true` for the ERC-8183 interface ID (to be assigned; placeholder `0x81830001`).

### Payment Handling

- **ETH payments**: `fund()` MUST be called with `msg.value == job.budget`. The contract MUST hold ETH in escrow until a terminal transition.
- **ERC-20 payments**: The caller MUST `approve()` the contract for `job.budget` before calling `fund()`. The contract MUST use `safeTransfer` / `safeTransferFrom` (OpenZeppelin SafeERC20 or equivalent).
- **Platform fees**: Implementations MAY deduct a platform fee from the budget on `complete()`. The fee percentage MUST be documented and SHOULD be bounded (e.g., ≤10%). The net amount paid to the provider and the fee amount SHOULD be emitted in `JobCompleted`.

### Evaluator Trust Model

The evaluator is the arbiter of job outcome. Implementations SHOULD support:

1. **Client-as-evaluator**: simplest case; client evaluates their own job.
2. **DAO / multisig**: decentralised governance controls `complete()`/`reject()`.
3. **Validation Registry** (ERC-8004 `IValidationRegistry`): on-chain ZK or TEE attestation of deliverable correctness.
4. **Dispute arbitration contract**: third-party arbitration (e.g., Kleros, UMA Optimistic Oracle).

The evaluator address is set at job creation and MAY NOT be changed after funding.

### ERC-8004 Integration (RECOMMENDED)

ERC-8183 implementations SHOULD integrate with ERC-8004:

- **Identity**: Provider SHOULD be an ERC-8004 registered agent. Implementations MAY reject providers with no registered `agentId`.
- **Reputation**: On `complete()`, implementations SHOULD call `giveFeedback(agentId, +delta, ...)` with tag `"task-completed"` on the ERC-8004 `IReputationRegistry`.
- **Reputation**: On `reject()` after `submit()`, implementations SHOULD call `giveFeedback(agentId, -delta, ...)` with tag `"task-rejected"`.
- **Reputation delta**: RECOMMENDED formula: `delta = budget / 1e15 / 2` (scaled to avoid uint overflow; implementations MAY choose any monotone function of budget).

## Rationale

### Why a separate ERC and not an extension of ERC-8004?

ERC-8004 covers agent identity and reputation; it deliberately excludes commerce mechanics to remain minimal. Separating commerce into ERC-8183 allows the two standards to evolve independently. ERC-8004 registries can be used by many different commerce protocols; ERC-8183 can support agents with or without on-chain identity.

### Why an evaluator role instead of client-only control?

Giving the client sole control over `complete()` creates a denial-of-payment attack: a malicious client simply never calls `complete()`. An independent evaluator role enables trust-minimised escrow. The `expiredAt` + `claimRefund()` mechanism provides a safety valve even if the evaluator goes dark.

### Why support both ETH and ERC-20?

ETH is the native currency of EVM chains and requires no approval flow. ERC-20 support is necessary for stablecoin payments (USDC, DAI), which agents may prefer for predictable revenue. Supporting both in a single interface avoids fragmentation.

### Why allow `provider = address(0)` at creation?

Many workflows require posting a job and accepting provider applications before committing to one. Separating `createJob` from `setProvider` supports open-bid, request-for-proposal, and auction flows (e.g., ERC-8183 can be combined with a commit-reveal auction at the application layer).

### Why include `deliverable` as a string rather than `bytes`?

String allows IPFS CIDs, HTTPS URLs, and on-chain calldata URIs to coexist without format overhead. Off-chain deliverable storage is intentional: on-chain content storage is too expensive for most work products. The deliverable URI/hash provides enough on-chain proof for reputation and dispute purposes.

### Gas considerations

- The `Job` struct is intentionally compact. `description` and `deliverable` are strings (dynamic), stored in contract storage only on write. Implementations concerned with gas SHOULD accept a `bytes32` content hash and emit the full URI only in events.
- The `reject()` path from `Open` status (client cancelling before funding) does not need a transfer; implementations SHOULD short-circuit.

## Backwards Compatibility

No backward compatibility issues. ERC-8183 is a new standard. It requires EIP-165 and RECOMMENDS ERC-8004, but neither dependency is mandatory for the core interface.

## Security Considerations

### Reentrancy
Implementations MUST use a reentrancy guard on all state-mutating functions that transfer ETH or tokens. The recommended pattern: update `job.status` to the terminal state **before** executing the transfer (checks-effects-interactions).

### Evaluator capture
A malicious or compromised evaluator can call `complete()` on fraudulent work or call `reject()` on good work. Mitigations:
- Use a multisig or DAO as evaluator.
- Use an on-chain validation registry (ERC-8004 `IValidationRegistry`) with cryptographic proof requirements.
- Time-bound evaluator power with `expiredAt` — after expiry, client can always reclaim funds regardless of evaluator.

### Griefing via expiredAt
A client could set `expiredAt` far in the future to prevent refund for a long time. Clients SHOULD set reasonable deadlines. Implementations MAY enforce a maximum job duration.

### Budget = 0
If `budget == 0` at the time `fund()` is called, the function MUST revert. This prevents "free work" scenarios where a provider submits work with no payment.

### Token approval race
ERC-20 approval and `fund()` are two transactions; a reorg could result in double-spend. Implementations SHOULD use `safeTransferFrom` with the exact `job.budget` amount to prevent over-spend.

### Integer overflow
All arithmetic (fee computation, reputation delta) MUST use Solidity 0.8.x built-in overflow checks or equivalent. Reputation delta computations SHOULD be capped to prevent underflow of registry values.
