## Abstract

Replace the BLS-signature-based RANDAO reveal with a hash-chain commit-reveal scheme. RANDAO's resistance to grinding currently relies on BLS signatures being *unique*, so that a proposer cannot bias its contribution. Since BLS relies on pre-quantum hardness assumptions, a quantum computer can recover a validator's key and predict its future reveals.

A hash chain depends only on preimage resistance, which is believed to be post quantum, and removes the dependency on the signature scheme entirely. Each validator commits to the tip of a generated hash chain. When proposing a block, the validator reveals the preimage of its currently stored commitment; the protocol verifies that the revealed value is indeed the preimage to the current stored commitment for that validator, mixes a per-validator personalization of the preimage into the RANDAO accumulator, and stores the preimage as the validator's new commitment.

Commitments are registered and changed via a new per-block-capped beacon operation (similar to `BLSToExecutionChange`). Validators that have not yet registered a commitment continue to use the legacy BLS reveal; this is a transitional path intended for removal in a later fork.

We again note the simplicity:  the protocol holds only the current commitment (32 bytes) and walks one link back per proposal for that validator.

## Motivation

The primary driver is post-quantum readiness.

Today's RANDAO contribution is a BLS signature over the epoch number. Its security rests on the *uniqueness* of BLS signatures: given the message and public key, exactly one valid signature exists, so a proposer cannot grind their own contribution to bias the randomness.

BLS is not post-quantum safe, so a cryptographically relevant quantum computer (CRQC) can recover a validator's secret key from its public key. Uniqueness still holds, but the attacker can compute the validator's reveals itself, predicting the chain's future randomness, and thus the proposer schedule, far in advance.

Most use cases for signatures like proposals, attestations, etc can move to any secure post-quantum scheme. RANDAO however, additionally depends on signature *uniqueness*, and most post-quantum schemes do not provide it allowing one to grind for a favorable randao contribution. The lean signature scheme currently planned for the consensus layer is grindable in exactly this way, since it includes a _salt_ component.

A hash chain sidesteps both problems in a simple way. First, there is no signature it's just chaining hashes together. In order to grind the hash chain, one needs to find a pre-image that favours the attacker with respects to the chosen hash function. Given our hash function is preimage resistant, this is believed to be hard even for a post-quantum attacker.

The commit-reveal structure also preserves RANDAO's existing security model, since the whole chain is fixed at commitment time, long before the validator knows its proposal slots or the mixes it might want to bias, so the proposer's only remaining lever is the same as the one it has today: reveal, or withhold and forfeit the block.

Note: This EIP does not make beacon chain randomness post-quantum secure end to end; block signatures and the registration operation's signature still use BLS. It incrementally introduces a post-quantum version of RANDAO, so that a later post-quantum fork which changes the signature-scheme and sets the initial hash-chain commitment at deposit time can complete the transition.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

These changes are applied to the consensus specifications at a fork to be scheduled.

### Hash chain construction (off-chain)

A validator generates a uniformly random chain secret `c_0` and computes a chain of length `n`:

```text
c_i = hash(c_{i-1})    for i in 1..n
```

where `hash` is SHA-256, as elsewhere in the consensus specifications. Note: the chain carries no validator identity, it is a simple hash chain.

Contributions are instead personalized per validator at mixing time (see Rationale). The chain secret `c_0` MUST NOT be the zero word: a zero reveal is rejected by `process_randao` since that value is used to indicate a validator that has not registered a hash chain.

The validator publishes `c_n` as their commitment and stores the chain (or the chain secret plus periodic checkpoints if chain is large). Reveals are consumed in reverse order: the first reveal is `c_{n-1}`, the next is `c_{n-2}`, and so on. Each revealed value becomes the new on-chain commitment, so the protocol requires no knowledge of `n` or of the validator's position in the chain.

Validators SHOULD choose `n` large enough such that a commitment change is never operationally necessary. `n >= 2**13` (~8,200 links) is RECOMMENDED. The whole can be generated in milliseconds and if one wanted to store all values, it would take up ~256KB to store. This exact value is of course arbitrary: the protocol never learns or enforces `n`, and since generation and storage stay cheap, most operators lose nothing by choosing a larger `n`.

#### Commitment Rotation

Independent of the initial sizing, client software SHOULD track how many links remain (the validator knows its own position in the chain, since it generated it) and, well before the low end is reached, automatically register a fresh commitment via `RandaoCommitmentChange`. 

The chain is exhausted only when its stored commitment reaches the secret seed `c_0`, whose preimage the validator does not hold. The trigger threshold therefore need only leave headroom for the replacement to activate: the new commitment becomes usable within `COMMITMENT_CHANGE_DELAY` while the old chain still serves proposals. 

Any modest threshold (a few hundred links, trivially cheap to over-reserve) makes the handover seamless and guarantees no proposal is ever missed due to the validator reaching the end of their chain. With the recommended `n` and this auto-renewal mechanism, running out of links is effectively impossible.

Note: A validator that rotates its commitment at or near a slot it proposes must bind the change to the correct post-reveal tip, since proposing walks the stored commitment and can make a change that is already in flight stale. The mechanics and the two ways to handle it are covered in the Rationale section.

### Block processing

#### Modified `process_randao`

```python
def process_randao(state: BeaconState, body: BeaconBlockBody) -> None:
    epoch = get_current_epoch(state)
    proposer_index = get_beacon_proposer_index(state)
    proposer = state.validators[proposer_index]
    if state.randao_commitments[proposer_index] != Bytes32():
        # Hash chain reveal [New in this EIP]
        assert body.hash_chain_reveal != Bytes32()
        assert hash(body.hash_chain_reveal) == state.randao_commitments[proposer_index] # check proposer knows preimage
        assert body.randao_reveal == G2_POINT_AT_INFINITY # bls randao reveal should be empty
        contribution = hash(body.hash_chain_reveal + uint_to_bytes(proposer_index)) # mix proposer index so contribution is unique
        mix = xor(get_randao_mix(state, epoch), contribution)
        state.randao_commitments[proposer_index] = body.hash_chain_reveal
    else:
        # Legacy BLS reveal
        assert body.hash_chain_reveal == Bytes32() # hash-chain reveal should be empty
        signing_root = compute_signing_root(epoch, get_domain(state, DOMAIN_RANDAO))
        assert bls.Verify(proposer.pubkey, signing_root, body.randao_reveal)
        mix = xor(get_randao_mix(state, epoch), hash(body.randao_reveal))
    state.randao_mixes[epoch % EPOCHS_PER_HISTORICAL_VECTOR] = mix
```

Note the two-layer structure of the hash-chain path:

- Verification checks the proposers chain step (`hash(reveal) == commitment`)
- The mix absorbs the *personalized* value along with the reveal value `hash(reveal + uint_to_bytes(proposer_index))`

This means that if a validator chooses the same commitment as another validator in an attempt to cancel out their contribution (due to xor), then this is not possible because what we add is the revealed value and the proposers index to the mix.

The non-zero assert protects the sentinel: a zero entry in `randao_commitments` means "unregistered", and the reveal is stored as the next commitment. Without the guard, a validator that committed to the zero word as a chain value would, upon revealing it, silently store the sentinel and flip onto the legacy branch, while its client, still believing itself registered, produced invalid blocks indefinitely. With the guard, the zero-revealing block is itself invalid: the validator misses proposals until it registers a fresh commitment. So "zero means unregistered" is an enforced invariant.

#### New `process_randao_commitment_change`

```python
def process_randao_commitment_change(state: BeaconState, signed_change: SignedRandaoCommitmentChange) -> None:
    change = signed_change.message
    assert change.validator_index < len(state.validators)
    assert change.new_commitment != Bytes32()
    validator = state.validators[change.validator_index]
    assert change.current_commitment == state.randao_commitments[change.validator_index]
    assert change.new_commitment != change.current_commitment
    domain = compute_domain(
        DOMAIN_RANDAO_COMMITMENT_CHANGE,
        genesis_validators_root=state.genesis_validators_root,
    )
    signing_root = compute_signing_root(change, domain)
    assert bls.Verify(validator.pubkey, signing_root, signed_change.signature)
    queue_randao_commitment(state, change.validator_index, change.new_commitment)
```

Called from `process_operations` for each element of `body.randao_commitment_changes`. This is the general commitment change path (initial registration, routine replacement, and recovery) and works for validators regardless of proposal schedule. The signing domain is computed against the genesis fork version (similar to `BLSToExecutionChange`) so that messages remain valid across forks.

The `current_commitment` binding however, makes every signed change single-use: applying the change falsifies the check on current_commitment, as does any subsequent proposal by the validator. A change message replayed from random parts in chain history is therefore invalid. This also just follows the same logic as `BLSToExecutionChange`.

#### New `queue_randao_commitment`

At most one commitment change per validator may be pending at any time; a block containing a change for a validator with an in-flight change is invalid.

```python
def queue_randao_commitment(state: BeaconState, index: ValidatorIndex, commitment: Bytes32) -> None:
    assert all(pending.validator_index != index for pending in state.pending_randao_commitments)
    state.pending_randao_commitments.append(PendingRandaoCommitment(
        validator_index=index,
        commitment=commitment,
        activation_epoch=get_current_epoch(state) + COMMITMENT_CHANGE_DELAY,
    ))
```

The resulting lifecycle, worked through to be more explicit: 

- A commitment change included in epoch `N` is queued with activation epoch `N + 3`. 
- The validator's previous commitment remains active throughout epochs `N` to `N + 2`: proposals in that window reveal from the old chain as normal. 
- At the epoch transition into `N + 3`, the pending entry is consumed and the new commitment becomes active. 
- A further change is includable from epoch `N + 3` onward (activating at `N + 6`). A change that was submitted during epochs `N` to `N + 2` would be invalid, since the first change is still in flight.

### Epoch processing

#### New `process_pending_randao_commitments`

Called from `process_epoch`, immediately after `process_registry_updates`. Pending commitments are applied in queue order once their activation epoch is reached. The one-pending-change-per-validator rule enforced in `queue_randao_commitment` guarantees entries never conflict.

```python
def process_pending_randao_commitments(state: BeaconState) -> None:
    next_epoch = Epoch(get_current_epoch(state) + 1)
    remaining = []
    for pending in state.pending_randao_commitments:
        if pending.activation_epoch <= next_epoch:
            state.randao_commitments[pending.validator_index] = pending.commitment
        else:
            remaining.append(pending)
    state.pending_randao_commitments = List[PendingRandaoCommitment, PENDING_RANDAO_COMMITMENTS_LIMIT](remaining)
```

### Gossip

A new global gossip topic `randao_commitment_change` carries `SignedRandaoCommitmentChange` messages. Unlike `bls_to_execution_change`, these messages are repeatable per validator, so explicit rate limiting is required. The rules are evaluated in order; in particular, the seen-this-epoch check precedes signature verification.

- **[REJECT]** `new_commitment` is zero, or `validator_index` is unknown.
- **[IGNORE]** a `SignedRandaoCommitmentChange` for `validator_index` has already been accepted in the current epoch.
- **[IGNORE]** a pending commitment change for `validator_index` already exists in the node's view of the state.
- **[REJECT]** `current_commitment` does not match the validator's `randao_commitments` entry in the node's view of the head state.
- **[REJECT]** the signature is invalid.

### Fork transition

At the fork epoch, `randao_commitments` is initialized with one zero entry per registry member and `pending_randao_commitments` is empty. Validators added to the registry after the fork likewise start unregistered (`add_validator_to_registry` is modified to append a zero entry) and use the legacy path until they register. All validators start on the legacy BLS reveal path and migrate by broadcasting a `SignedRandaoCommitmentChange`. At `MAX_RANDAO_COMMITMENT_CHANGES = 128` per block, the full current validator set (~1M) can register in under two days of full blocks; there is no deadline, and unregistered validators simply continue on the legacy path.

### Sunset of the legacy path

The BLS reveal branch in `process_randao`, and the `randao_reveal` field itself, are transitional and SHOULD be removed in a later fork after registration has saturated, naturally the fork that reworks deposits for post-quantum signatures, at which point initial hash chain commitments move into the validator onboarding flow and the `RandaoCommitmentChange` signature migrates to the post-quantum scheme. 

Note: the `RandaoCommitmentChange` operation itself is permanent and will be used as the recovery mechanism for validators that lose their chain secret or need to generate a new hash chain.

## Rationale

### Why the mix absorbs `hash(reveal + validator_index)`

The mixed-in contribution must satisfy two requirements:

- *unpredictability*: the mix must never absorb a value derivable from pre-reveal public state(meaning it can't be predictable by everyone). This rules out the legacy shape `hash(reveal)`. For a hash chain, that is by definition the validator's stored public commitment.
- *per-validator distinctness*: hash-chain commitments are naively copyable, so the mix must not absorb the same word when two validators reveal the same value. This rules out mixing the raw reveal, which would let a validator that registered a copy of another's commitment cancel the victim's contribution out of the xor accumulator, if they are both in the same epoch.

`hash(reveal + uint_to_bytes(validator_index))` satisfies both conditions: it is computable only if you know `reveal`, and two validators revealing the identical value contribute distinct entropy to randao, since the validator index is unique per validator.

### Why xor is retained rather than a hashing accumulator

Every element under the xor operation is it's own inverse, so re-injecting a value cancels it out of the mix. Under BLS reveals, this was not possible because a reveal had to be a valid signature by the proposer's own public key, so no party could ever inject another parties contribution. Naively replacing BLS with a hash chain, dropped this "ownership" property that made xor safe since a hash chain commitment is trivially copyable. We fixed this as seen by adding thr validators index to their contribution, ensuring that each contribution to he mix could only have been made by a particular validator, even if two validators had the same hash chain.

An alternative would be to change the accumulator itself to `mix = hash(mix + reveal)`. A hash accumulator has no efficiently computable inverses, so a duplicated reveal degrades from cancelling the victim's contribution to being merely predictable (a null contribution).

Future versions of this EIP, may switch to this variant allowing us to remove the proposer index from the randao. xor has a cleaner argument with respects to randao; the mix is uniform if any single contribution to it is uniform and independent, theres no assumption on a hash function being a random oracle here.

### Why contributions are personalized at mixing rather than chains being identity-bound

Commitments are public state, and pure hash chains carry no identity, so nothing stops a validator from registering a value it did not generate, including a *copy of another validator's current commitment*, via a perfectly fresh, correctly signed change of its own. If the mix absorbed the raw reveal, that copy would arm a cancellation attack. 

Attacker `W` registers victim `V`'s commitment `c`. Until `V` reveals, `W` holds a commitment it has no preimages for and misses every proposal it draws. But once `V` reveals `p`, the value is public and satisfies `W`'s stored commitment too: `W`'s next proposal validly reveals `p`, and since xor is self-inverse, re-injecting `p` would cancel `V`'s contribution out of the accumulator. The attack would also be self-sustaining: after revealing, both validators' stored commitments equal `p`, so `W` shadows `V`'s chain link-for-link indefinitely. This is strictly stronger than withholding or degenerate chains, which can only null *one's own* contribution: it erases someone else's, attacking RANDAO's actual security assumption that one honest contributor per mixing period suffices.

The natural alternative binds identity into every chain link (`c_i = hash(c_{i-1} + pubkey)`), making a copied commitment *unusable*: the victim's revealed preimage verifies only under the victim's key. This works, but places the defense in validator-side construction discipline. Every client and tool must weave the identity into chain generation; a chain built the textbook way silently fails on its first reveal (exactly the erroneous-commitment liveness trap); chains cannot be generated independently of key material; and the identity must be the *pubkey* (the only identifier that exists before deposit), creating a permanent requirement that BLS pubkeys remain resolvable for as long as any bound chain lives.

Personalizing at mixing time enforces the same guarantee in one line of protocol-controlled code that no validator can misconstruct. A copied commitment remains *usable* (the copier can propose once its host has revealed) but harmless: the two contributions differ by construction, so no cancellation is possible even for identical reveals, and the copier's contribution is merely predictable: the degenerate-chain null case, bounded by withholding. The residual oddity is a validator free-riding on another's chain, proposing without holding any secret; this costs the network nothing, and costs the free-rider its reliability (it misses whenever it outpaces its host's reveals).

The validator index, not the public key, is the personalization tag. Identity is needed only at mix time, when the state is at hand, so pre-deposit availability (the constraint that forced the pubkey in the bound-chain design) is moot. The index is smaller (a 40-byte hash input fits one SHA-256 compression block where the pubkey's 80 bytes take two), permanent, never reused, and credential-agnostic: neither chains nor their verification depend on the validator's signature scheme in any way, so a future post-quantum credential migration touches nothing in this EIP. A registration-time uniqueness check on commitments was also considered and rejected: it would require a global index of all commitments in state, and with personalization in place, duplicate commitments are harmless anyway.

### Why one link per block proposed

The legacy reveal is a signature over the epoch, so two proposals by the same validator in one epoch reveal the same value. Consuming one link per block is simpler due to no per-epoch book-keeping, makes every block contribute fresh entropy, and costs nothing given that chains are cheap to generate at any reasonable length.

We also note that with consolidations, the probability of a validator proposing multiple times in the same epoch increases, effectively cancelling out their randao.

### Why the protocol does not track chain length

Storing only the current commitment and requiring a preimage per proposal makes the chain length a purely private, off-chain parameter. The protocol state cost is one 32-byte entry per validator regardless of chain length, and validators can size chains so that a commitment change never becomes operationally necessary. The only reason to store the chain length is if we wanted to mandate that every validator generated a chain of a specific length, which seems unnecessarily restrictive and adds complexity.

### Why an activation delay on commitment changes, and why `MIN_SEED_LOOKAHEAD + 2`

The scheme's grinding resistance comes entirely from the commitment predating the validator's knowledge of what it would want to bias. If a validator could change their commitment just before proposing and it became active instantly, they could grind for a value that would be favourable for them. We add a delay and the delay is sized so the changer cannot even know *whether* they propose in the activation epoch at the time they choose the chain. 

Duties for epoch `E` are drawn from the mix as of the end of epoch `E - MIN_SEED_LOOKAHEAD - 1`. A delay of `MIN_SEED_LOOKAHEAD + 1` almost achieves this but leaks at the epoch boundary: a change included in the *last slot* of epoch `N` would activate at `N + 2`, whose duties derive from the end-of-`N` mix, essentially fully determined at that inclusion moment. With `COMMITMENT_CHANGE_DELAY = MIN_SEED_LOOKAHEAD + 2`, a change included anywhere in epoch `N` activates at `N + 3`, whose duties depend on the end-of-`N + 1` mix, unknowable during `N` because all of epoch `N + 1`'s reveals are still secret. The invariant is then unconditional: a chain is always fixed strictly before the seed that draws any duty it could ever serve.

An alternative design was to forbid commitment changes by validators with pending proposal duties. It however requires duty-awareness logic which complicates gossip and is more complex than the duty agnostic scheme.

### Why at most one pending change per validator

Allowing multiple pending entries per validator, with later entries overwriting earlier ones as they activate, would make redundant changes *valid*, which enables queue stuffing: a proposer could fill the entire `MAX_RANDAO_COMMITMENT_CHANGES` budget with changes for a single validator, costing the network 128 signature verifications and 128 queue entries for a single state effect, and crowding legitimate registrations out of the shared per-block budget. 

It would also undermine gossip rate limiting: the "IGNORE if a pending change exists" rule is only sound if a further change for that validator is guaranteed meaningless, which overwrite semantics contradict. With the one-pending rule, every included change corresponds to a distinct validator making a meaningful transition, and duplicates are invalid rather than merely wasteful.

### Why a change binds the commitment it replaces

Without the `current_commitment` predicate, a signed change would be valid forever, and replaying one from chain history would essentially be an attack. The stored commitment walks down the chain with every proposal, publishing each consumed link: after a validator registers `c_n` and proposes `k` times, the values `c_{n-1} … c_{n-k}` are public in block history. Replaying the original registration would reset the commitment to `c_n`, making the validator's next `k` contributions publicly predictable, collapsing the "unknown co-contributions" wall for that validator, and at scale allowing mixes and duties to be predicted beyond the lookahead horizon. Each replay would also occupy the validator's single pending slot, letting an attacker grief legitimate re-registration.

The `current_commitment` binding is the analogue of the idempotence predicate that makes `BLSToExecutionChange` replay-safe (its handler asserts the credentials are still BLS-prefixed, which the first application falsifies). Here the predicate is stronger than single-use: a signed change is invalidated not only by its own application but by any subsequent proposal, since proposing rewrites the stored commitment. Stale messages fail a REJECT-level gossip check, incurring peer penalties rather than consuming rate-limit budget.

This has a concrete operational consequence for a validator that wants to rotate its commitment at or near a slot it proposes. Let its stored commitment be `c_i`. A proposal reveals `c_{i-1}` and walks the stored value `c_i` to `c_{i-1}`, and a change is valid only while its `current_commitment` matches whatever value is stored at the moment the change is processed. Three cases follow:

1. **Gossip, then propose before inclusion.** A change broadcast with `current_commitment = c_i` goes stale the instant the validator proposes and walks the tip to `c_{i-1}`. Nodes that validate it afterward fail the `current_commitment` check and issue a REJECT, descoring the honest peers that relayed a message which was valid when they sent it.
2. **Self-inclusion (the clean path).** A proposer can carry its own change in its own block's operations list. `process_block` runs `process_randao` before `process_operations`, so the reveal walks `c_i` to `c_{i-1}` first and the change is then checked against `c_{i-1}`, a value the proposer knows at build time. It signs `current_commitment = c_{i-1}` (the tip it reveals in that block), and the change lands atomically with no gossip race.
3. **Gossip fallback.** A validator that already gossiped a change and then proposes SHOULD re-sign it with `current_commitment` advanced to the now-walked tip `c_{i-1}`, leaving `new_commitment` unchanged.

The trade-off is that a *pre-signed* change goes stale whenever the validator proposes, which weakens the pre-signed-recovery pattern familiar from voluntary exits. In practice recovery still works: a validator that has lost its chain secret stops proposing, so its stored commitment stops moving and a recovery message signed against it stays valid. The alternative (a per-validator `commitment_nonce` counter in state, bumped only by changes so pre-signed messages survive proposals) was rejected as not worth the extra state for a marginal operational convenience, but is the natural upgrade path if durable pre-signed recovery becomes a requirement.

Relatedly, the handler also rejects a no-op change whose `new_commitment` equals its `current_commitment`. Refreshing to the same tip is never useful: it adds no runway, which is extended only by committing to a fresh chain, not by re-signing the current tip. At best such a change is pure cost (a signature verification, an operation slot, and the validator's single pending slot held for `COMMITMENT_CHANGE_DELAY`); at worst, if the validator proposes during the delay and walks its stored commitment down, activation resets the commitment back to the signed tip and rewinds the validator onto already-revealed links, a self-inflicted form of the replay-rewind above. The `new_commitment != current_commitment` assert removes the class outright.

### Why keep a legacy fallback instead of a registration deadline

There's currently no urgent rush to have validators be fully post quantum, so the benefits of adding a deadline here are not obvious. Validators will use BLS anyways. Moreover, there will likely be a long tail of validators whom do not submit the message because they are running custom software. We there defer the forced cutover to a fork that must touch these code paths anyway; ie the post quantum switchover fork, while allowing us to test out the codepaths.

## Backwards Compatibility

This EIP requires a scheduled consensus-layer hard fork. Within the fork, the change is backwards compatible from the validator's perspective: unregistered validators continue proposing exactly as today. Downstream consumers of `randao_mixes` (including the execution layer's `PREVRANDAO`) are unaffected; the mix remains a 32-byte accumulator updated once per block, only the provenance of contributions changes.

## Security Considerations

### Biasability is unchanged

The proposer's only degree of freedom is still withholding: reveal and propose, or withhold and forfeit the block plus its rewards. This is the same one-bit-per-proposer bias RANDAO has today, with the same economic cost. The chain being validator-generated (unlike a BLS signature, which the validator cannot choose) does not add grinding power, because the entire chain is fixed before the validator knows its proposal slots or the co-contributions to any future mix.

### Commitment changes cannot be used reactively

A validator who sees a proposal duty approaching cannot swap chains to influence the outcome. Any change signed after duties are known activates `COMMITMENT_CHANGE_DELAY` epochs after inclusion, strictly after every slot for which duties were knowable at signing time. 

Timing games around inclusion add nothing: delaying a change's broadcast shifts *which* pre-committed chain is active at a future slot, but choosing between two values that were both fixed before the target mix was predictable confers no bias. 

Nor can changes be replayed by third parties to rewind a validator onto already-revealed chain segments: the `current_commitment` binding makes every signed change single-use.

### Degenerate chains

A validator may choose pathological values for its *own* chain, for example deriving the chain secret from a publicly known constant like the genesis hash, so that all its reveals are predictable in advance and contribute no effective entropy. 

This is not possible to exhaustively restrict and we do not because RANDAOs security does not assume that *every* individual's contribution is honest. If this were the case, then a validator choosing to add nothing by witholding would also break the security; one honest contributor per mixing period suffices.

We do forbid the pathological case of using the zero word, since this would collide with the unregistered sentinel when revealed, so `process_randao` rejects zero reveals.

The bounded-by-withholding argument extends to *copied* chains because contributions are personalized at mixing: a validator that registers a copy of another's commitment and replays the victim's exposed preimage contributes `hash(reveal + uint_to_bytes(its own index))`: predictable, but distinct from the victim's contribution, so it cannot cancel anything out of the xor accumulator.

### Chain loss and theft

Losing the chain secret makes a validator unable to produce valid blocks until it registers a fresh commitment via `RandaoCommitmentChange`; the failure mode is missed proposals, not slashing risk.

Theft of the chain secret alone lets an attacker predict (not choose) the validator's future contributions, strictly less power than theft of the signing key today. Validator clients SHOULD treat the chain secret with the same custody standards as signing keys. One should not derive it in such a way that it links to their signing key as a safety measure because the hash chain if taken to the limit, can be revealed, whereas ones secret key should never be revealed.

### Reveal exposure in orphaned blocks

The legacy BLS reveal signs the epoch number, so a reveal exposed in a block that never makes the canonical chain is worthless outside that epoch. A hash-chain reveal has no such time binding: a reveal published in an *orphaned* block remains the validator's mandatory next contribution at whatever future slot it next proposes. An observer who collects reveals from orphaned blocks (or losing forks) accumulates known future co-contributions.

The mitigation uses existing machinery: a validator whose block is orphaned after broadcasting SHOULD submit a commitment change, retiring the exposed chain.

The exposure window is then bounded by inclusion time plus `COMMITMENT_CHANGE_DELAY`, roughly 3 to 4 epochs. One risk is that the validator draws another proposal inside that window, before the replacement activates. This is the same risk we have if a validator with the BLS regime re-proposes in the same epoch.

### Erroneous commitments

Preimage possession is unverifiable at registration: the protocol cannot distinguish a commitment whose chain the validator holds from a typo or the output of a buggy key-derivation.

The consequence of registering a bad commitment is missed proposals from its activation until a correction activates: at least `COMMITMENT_CHANGE_DELAY` epochs even if caught immediately. This failure mode does not exist under BLS reveals, where there is nothing to misconfigure beyond the signing key itself. Validator clients SHOULD verify a commitment by walking the full chain before signing a change.

### Quantum adversaries

The reveal verification relies only on SHA-256 preimage resistance, conjectured to retain ~64-bit security against quantum attack. The transitional BLS reveal path and the BLS signature on `RandaoCommitmentChange` remain quantum-vulnerable, as do block signatures generally; this EIP removes RANDAO's structural dependence on signature uniqueness so that the eventual post-quantum fork is a signature-scheme swap rather than a randomness redesign too.

### Denial of service via commitment changes

Repeated commitment changes incur no randomness advantage, so spamming commitment changes is a possible DOS vector. Here are the ways we mitigate them:

- **Blocks.** The operations list is capped at `MAX_RANDAO_COMMITMENT_CHANGES` per block (at most 128 additional BLS verifications).
- **Gossip.** Commitment changes are repeatable, so unlike `bls_to_execution_change`, first-seen-per-validator deduplication is insufficient: a single validator could otherwise sign unbounded *distinct* valid messages and force a signature verification for each. The gossip IGNORE rules cap this at one verification per validator per epoch. Non-validators cannot participate at all: their messages fail signature validation and incur gossipsub peer penalties. Replayed historical messages fail the `current_commitment` REJECT check (also penalized), so they cannot be used to occupy a validator's per-epoch IGNORE slot and censor its genuine change.
- **State.** `queue_randao_commitment` enforces at most one pending change per validator, and the queue is additionally bounded by inclusion rates: at most `MAX_RANDAO_COMMITMENT_CHANGES` entries per block, each resident for `COMMITMENT_CHANGE_DELAY` epochs, giving a worst-case steady state of ~12,300 entries.
