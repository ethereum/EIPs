## Abstract

This proposal surcharges writes to long-untouched state. It defines a single, governance-chosen `CUTOFF_BLOCK`. Any account or storage slot whose [EIP-8188](./eip-8188.md) `last_written_block` is below the cutoff is **Inactive** and costs more to write. Everything else is **Active** and priced as today. Inactive state stays in the trie and the state root, so there is no eviction, witness, or resurrection mechanism.

The proposal also recommends that clients converge on a single shared file format for the inactive set, so the cold state can be held outside the hot database and distributed out-of-band rather than only over P2P.

## Motivation

State keeps growing, and most of it is cold: written once and never touched again. That cold state still sits in the working set every node maintains, and the cost of mutating it is the same flat price as mutating state that was written yesterday.

A companion proposal prices this with rolling renewal-age periods. This proposal takes the simplest possible form of the same idea: a single cutoff block, chosen by social coordination, that splits state into Active and Inactive once. There are no periods, no anchor, and no ongoing reclassification. The only parameter that matters is one number that the community agrees on.

A single static boundary also pairs naturally with archiving. The Inactive set is fixed at activation, so it can be serialized once into immutable files. If clients agree on the same file format, those files become interchangeable: a node can drop the cold set from its hot database, hold it as files, and sync it from any peer or mirror without re-deriving it. This mirrors how Ethereum already distributes block history through the [E2Store](https://github.com/eth-clients/e2store-format-specs) family of era files.

## Specification

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119) and [RFC 8174](https://www.rfc-editor.org/rfc/rfc8174).

This proposal reads the `last_written_block` field defined in [EIP-8188](./eip-8188.md). [EIP-8188](./eip-8188.md) owns the encoding and the rules that update that field. This proposal only reads it to classify state.

### Parameters

| Name | Value | Description |
| --- | ---: | --- |
| `CUTOFF_BLOCK` | TBD | Block number below which state is Inactive. Chosen by social coordination, after [EIP-8188](./eip-8188.md) activation |
| `INACTIVE_ACCOUNT_WRITE` | TBD | Account mutation cost for Inactive state |
| `INACTIVE_STORAGE_WRITE` | TBD | Storage write cost for Inactive state |

`INACTIVE_ACCOUNT_WRITE` and `INACTIVE_STORAGE_WRITE` MUST exceed the corresponding flat write cost charged for Active state.

### Tier Determination

An account or storage slot is classified from its `last_written_block`:

```python
if last_written_block < CUTOFF_BLOCK:
    tier = INACTIVE
else:
    tier = ACTIVE
```

Items that [EIP-8188](./eip-8188.md) decodes with `last_written_block = 0` (never written since its activation) fall below any sensible `CUTOFF_BLOCK` and are therefore Inactive.

### Tiered Write Costs

The tier determines the gas cost for mutations of state that already exists in the trie. Active mutations keep their current flat write cost. Inactive mutations use the surcharged constants. Read operations are unaffected.

| Operation | Active | Inactive |
| --- | --- | --- |
| Account mutation | current cost | `INACTIVE_ACCOUNT_WRITE` |
| Storage mutation | current cost | `INACTIVE_STORAGE_WRITE` |

The tier MUST be determined from `last_written_block` as it stands at the start of the operation, before [EIP-8188](./eip-8188.md) sets it to the current block number. A write to Inactive state therefore pays the surcharge once. Because `CUTOFF_BLOCK` is fixed and the write sets `last_written_block` to a value at or above it, the item becomes Active for all later writes.

### State Creation Is Not Tiered

The tiers apply only to mutations of existing state. State creation is never classified as Inactive and never pays the surcharge. Writing to a never-before-initialized account, and a storage slot transitioning from zero to non-zero (including a slot previously cleared to zero and later restored), are creation rather than tiered writes.

State creation is priced by [EIP-8037](./eip-8037.md). To keep the schedule consistent, its cost MUST NOT be lower than the Inactive write cost.

### Disambiguation from EIP-2929 warm/cold

Active and Inactive are write tiers determined by `last_written_block` relative to `CUTOFF_BLOCK`. They affect write costs only. [EIP-2929](./eip-2929.md)'s warm/cold distinction is a within-transaction first-touch concept that continues to apply to reads as before. For a write, first determine the tier, then apply the corresponding write cost.

### Shared File Format for the Inactive Set

This section is RECOMMENDED, not consensus. Inactive state remains in the trie and the state root, so nothing here changes block validity.

Clients SHOULD converge on a single shared file format for the Inactive set as of `CUTOFF_BLOCK`. The goal is interchangeable files, not a particular encoding. Such a format SHOULD be:

- **Immutable.** `CUTOFF_BLOCK` is fixed, so the Inactive set as of that block does not change and the files can be written once and shared.
- **Self-verifying against the state root.** The contents are a subset of the state at `CUTOFF_BLOCK`, so both membership (each item's stored `last_written_block`) and values are provable against that state root. A client MUST be able to import the files without trusting their producer.
- **Indexable.** A client SHOULD be able to look up an account or slot without scanning the whole archive.

The [E2Store](https://github.com/eth-clients/e2store-format-specs) family already used for block history is the natural basis for such a format. The concrete encoding is left to a separate standardization effort.

With an agreed format, a client MAY drop the Inactive set from its hot database and hold it as files, syncing it out-of-band rather than over P2P. A client MUST still hold an item before writing it, because a write recomputes the state root.

## Rationale

### Why a single cutoff instead of rolling periods?

A companion proposal buckets write-age into rolling periods so that state ages into the Inactive tier continuously. That requires period parameters and an anchor, and every item's tier moves over time. A single cutoff removes all of that. There is one number to agree on, the classification is a single comparison, and the Inactive set is fixed at activation.

The cost is that newly cold state does not age in on its own. Once state written before `CUTOFF_BLOCK` is the Inactive set, state written after it stays Active forever under this EIP. Re-tightening the boundary is a later social decision: a future EIP advances `CUTOFF_BLOCK`. This is a deliberate trade of automatic coverage for simplicity and explicit governance.

### Why keep Inactive state in the state root?

Evicting Inactive state would let nodes delete it, but it would also pull in the hard parts of state expiry: witnesses to revive expired state, resurrection-conflict handling, and often address-space changes. Those are the problems that have stalled full state expiry. Keeping Inactive state in the root avoids all of them. The proposal still shrinks each node's hot working set, because a client can move the cold set to files and out of the database it touches every block, while the data remains provable against the state root.

### Why surcharge writes?

The surcharge aligns the cost of dragging cold state back onto the hot path with the work it causes. A node that has moved the Inactive set to files has to load an item back before it can be written. Pricing that path above an ordinary write discourages gratuitous churn of long-dead state without forbidding it.

### Why a shared file format?

The value is interoperability. If every client writes the Inactive set the same way, the files are interchangeable and can be distributed over ordinary channels such as HTTP and BitTorrent, the same way [E2Store](https://github.com/eth-clients/e2store-format-specs) era files distribute block history. A node can then obtain the cold set from any mirror instead of re-deriving it, which makes dropping it from the hot database practical. Leaving the byte-level encoding to a separate effort keeps this proposal focused on the consensus surcharge.

## Backwards Compatibility

This is a backwards-incompatible gas repricing that requires a scheduled network upgrade. It depends on [EIP-8188](./eip-8188.md) and MUST activate at or after the same fork. `CUTOFF_BLOCK` MUST be chosen above [EIP-8188](./eip-8188.md)'s activation block so that `last_written_block` is meaningful for the comparison.

Wallets and RPC providers MUST update `eth_estimateGas` to charge the surcharge when a write targets Inactive state.

## Security Considerations

### Write-inactivity is not read-inactivity

The cutoff classifies state by write recency only. A storage slot can be read on every block yet never rewritten, leaving it Inactive. The classification MUST NOT be read as "cold" for access purposes. A client that moves Inactive state to slower storage or to files SHOULD account for read frequency separately, since placing frequently read Inactive state on a slow path could become a denial-of-service vector.

### Builder availability

Because Inactive state stays in the state root, writing it recomputes the root and requires the item's data. A block builder that has dropped the Inactive set to files MUST reload the relevant items before including a transaction that writes them. A builder that cannot reload in time may be unable to include such a transaction, which is a liveness consideration for clients that aggressively offload cold state.

### Choosing the cutoff

`CUTOFF_BLOCK` is a governance choice. Setting it too close to the present makes a large fraction of recently used state Inactive and over-penalizes ordinary activity. Setting it too far in the past leaves little cold state behind the cutoff and blunts the effect. The value SHOULD be derived from the observed write-age distribution of state at the time of the fork.
